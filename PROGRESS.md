A one-paragraph entry per completed phase. Newest first. Source of truth for

## Conventions

- Each entry: phase number + title, completion date (UTC), commit range, and a
  short summary of what landed and what was deliberately deferred.
- "Deferred to Phase N" is preferred over "TODO" — every gap should have a
  named home further down the plan.
- If a phase ships partial scope, mark it `partial` and list the missing
  Definition-of-Done items so the next session knows what to pick up.

## Phase 12 — Notifications

_Status: complete (2026-05-19). Branch: `phase/12-notifications`. Commit range: `93189bf..749c509` (11 commits)._

What landed:

- **`@dankdash/notifications` package** (`packages/notifications/`) — Pure, framework-free package the API + workers can both import without dragging NestJS or Drizzle in. Three concentric layers: (1) **types** (`src/types.ts`) declare `NotificationChannel = 'push' | 'sms' | 'email' | 'in_app'` and the closed `NotificationTemplateKey` union — 17 keys spanning consumer order lifecycle, driver dispatch, vendor payouts/discrepancy, auth onboarding — plus the discriminated `Recipient`, `RenderedNotification`, and `ProviderSendResult` shapes. (2) **templates** (`src/templates/`) keyed by template key, each a pure function that takes a typed payload and returns one rendered notification per applicable channel. (3) **providers** (`src/providers/`) — thin adapters over APNs/Twilio/Resend that take a `RenderedNotification` + `Recipient` and produce a `ProviderSendResult`. Vitest config gates 100% line+branch+function+statement coverage on src; barrels, type-only modules, and tests are excluded.

- **17 templates with format helpers + 100% coverage** (`packages/notifications/src/templates/`) — Files: `auth.ts` (welcome, id_verification_required), `dispatch.ts` (offer, offer_expired, canceled), `dispensary.ts` (new_nearby), `order-lifecycle.ts` (accepted, prepping, ready, picked_up, arriving, arrived, completed, payment.failed, refund.issued), `vendor.ts` (payout.completed, metrc.reconciliation_discrepancy), plus the `registry.ts` map that wires every key to its template. Pure formatters in `format.ts`: `formatUsdCents` (centses to `$X,XXX.XX`), `formatMilesShort` (decimal mi with one digit + " mi" suffix), `formatMinutes` ("less than a minute" / "1 minute" / "N minutes"), `formatOrderShort` (`#01935F3D` from a UUIDv7). Each template returns the right channel set per type: order.accepted → push+sms+in_app; order.completed → push+email+in_app; payment.failed → push+sms+email+in_app; vendor.\* → email-only. 17/17 templates × snapshot stability tests + format edge cases (IEEE-754 `12.95.toFixed(1)` gotcha documented inline).

- **APNs/Twilio/Resend providers + 100% coverage** (`packages/notifications/src/providers/`) — Each provider is an `Injectable`-shaped class with a `channel` getter, a `send(recipient, rendered)` method, and a `shutdown()` for graceful drain. **`ApnsProvider`** wraps `@parse/node-apn` v8 via a narrow `ApnsProviderHandle` interface so tests inject a fake without spinning up a real `apn.Provider`. Constructs `apn.Notification` per token; sets `topic` from `recipient.bundleId` (falls back to `defaultTopic`), `priority`/`pushType` for silent vs. alert sends, `collapseId` and `contentAvailable` when the template requests them. Failure classifier maps the documented dead-token reasons (`BadDeviceToken` / `Unregistered` / `DeviceTokenNotForTopic`) to `{ ok: false, retireApnsToken: <token> }` so the dispatcher can flip the matching `push_tokens.is_active` to false; `PayloadTooLarge` is `{ retryable: false }`; everything else stays retryable. **`TwilioSmsProvider`** wraps Twilio v5 via a `TwilioMessagesApi` narrow interface; sender selection prefers `messagingServiceSid` over `fromNumber`, throws `TypeError` at construct time if neither is set (or both are empty strings). Permanent error codes (`21211`, `21610`, `21614`, `21408`, `30003`, `30005`, `30006`, `30007`) map to `retryable: false`; everything else (including SDK transport-level throws without a code) stays retryable. **`ResendEmailProvider`** wraps Resend v4 via `ResendEmailsApi`; `fromOverride` on the rendered payload wins over the constructor default; `html` is only included in the payload when the template provided one (must NOT be `html: undefined` — Resend rejects the request). Permanent error names (`invalid_to_address`, `invalid_from_address`, `validation_error`, `missing_required_field`, `invalid_attachment`, `restricted_api_key`) map to `retryable: false`. All three providers throw `TypeError` on wrong-channel recipient or rendered payload (caller bug, not a delivery failure). 55 provider tests total (18 APNs + 21 Twilio + 16 Resend) at 100% coverage including the real `apn.Provider` constructor branch reached via `vi.resetModules() + vi.doMock + dynamic import`.

- **`NotificationsRepository` + `push_tokens` table** (referenced from `@dankdash/db`, existing from Phase 1 schema) — The dispatcher writes one `notifications` row per channel per attempt with `payload`/`templateKey`/`channel`/`sentAt`/`providerRef`/`error` — the row IS the audit trail and nobody else writes to that table. The iOS in-app inbox surface (`/me/notifications`, deferred to Phase 13) reads the same rows. `push_tokens` is keyed `(user_id, device_id, app_variant)` so the iOS client may re-register on every cold launch without producing duplicates; `is_active = false` is the only "deletion" — the dispatcher's APNs-token-retire path and the user-facing DELETE both flip the flag rather than hard-delete.

- **`/v1/me/push-tokens` endpoints** (`apps/api/src/modules/notifications/push-tokens.{controller,service}.ts` + `dto/push-token.dto.ts`) — `POST /v1/me/push-tokens` upserts a token+platform+appVariant tuple; `DELETE /v1/me/push-tokens/:id` soft-deactivates (returns 204 even for an already-deactivated row so iOS retry-on-network-flake doesn't surface a 4xx). Both endpoints are JWT-guarded via the global `JwtAuthGuard` + `@Roles('customer', 'driver')` — vendor staff and admin operators don't carry APNs tokens. Cross-user deactivation surfaces as 404 (same as "does not exist") so a probing client cannot enumerate token ids. Rate-limited per-user (30 req/min) since a single account hammering the register endpoint (broken iOS retry loop) is the threat shape. 30 controller+service tests cover the upsert idempotency, the cross-user 404, the already-deactivated 204, and the rate-limit envelope shape.

- **`RedisNotificationDedupeStore`** (`apps/api/src/modules/notifications/notification-dedupe.store.ts`) — `acquire(key, ttlSeconds)` wraps `SET key value EX ttl NX` (the atomic op — anything that splits SET + EXPIRE risks orphaned keys on a mid-flight crash). Returns true iff this call acquired the key (i.e., was not already set). The dispatcher holds the key for 24h per spec §5.2 — the "idempotency window" for retries from the order machine + webhook bus. EX seconds (not PX ms) keeps the key visible in any Redis dashboard with second precision. Interface abstracted so test composition can swap in an in-memory store without touching Redis. 9 tests cover the SET NX semantics, key prefix isolation, and the failed-acquire branch under a duplicate-event replay.

- **`NotificationDispatcher`** (`apps/api/src/modules/notifications/notification-dispatcher.service.ts`) — The single fan-out chokepoint every notification event flows through. Per-dispatch: (1) acquire the Redis dedup key `${userId}:${templateKey}:${idempotencyKey}` for 24h; bail on failed acquire (duplicate fired). (2) Load the user; if missing, skip with `user_not_found`. (3) `renderTemplate(key, payload)` once; iterate the array of rendered notifications. (4) Per channel: insert a `notifications` row up front so every attempt is auditable, resolve the recipient (push: active tokens from `push_tokens.listActiveForUser(userId, appVariant)`; sms: `users.phone`; email: `users.email`; in_app: trivially the userId), then either persist (in_app — the row IS the artifact) or hand to the provider. (5) On provider success: `notifications.markSent(rowId, providerRef)`. On provider failure: `notifications.markFailed(rowId, error)` plus, when the result carries `retireApnsToken`, call `pushTokens.deactivateByApnsToken(token)` to flip the matching row. (6) Channels with no configured provider (e.g. SMS disabled in dev) produce a `provider_unavailable` row so the gap is visible to ops rather than silently dropping. Inline by design — no separate outbox worker, because the iOS app subscribes to `notifications` via realtime push and the row insertion IS the consumer-visible artifact. 12 dispatcher tests cover dedup-hit short-circuit, missing-user short-circuit, multi-channel fan-out, provider-unavailable path, recipient-missing path, APNs-token-retirement plumbing, and the `markSent`/`markFailed` audit envelope under success+failure mixes.

- **`OrderNotificationsListener`** (`apps/api/src/modules/notifications/order-notifications.listener.ts`) — `@OnEvent(ORDER_TRANSITIONED_EVENT, { suppressErrors: true })` subscriber over the post-commit emit from `OrderTransitionService`. State→template map: `accepted` → `order.accepted`; `prepping` → `order.prepping`; `ready_for_pickup` → `order.ready`; `picked_up` → `order.picked_up` (resolves driver firstName via drivers+users repos); `arrived_at_dropoff` → `order.arrived` (same driver lookup); `delivered` → `order.completed` (with totalCents); `payment_failed` → `payment.failed` (with amountCents + decline reason). Idempotency key per dispatch is `${orderId}:${toStatus}` so a duplicate `ORDER_TRANSITIONED_EVENT` for the same transition reaches the Redis SETNX inside the dispatcher and bails before any provider is touched. Listener catches every thrown error and logs at error-level — the transition itself is already durable, the dispatcher writes the failure row, and we must not propagate back into the discard path of `EventEmitter2.emit`. States with no entry (`awaiting_driver`, `driver_assigned`, `en_route_*`, terminal `canceled`/`rejected`/`returned`/`disputed`) are intentional no-ops (dispatch handles its own side, terminal states are either operator-driven or silent). 18 listener tests cover every mapped state, the missing-driver short-circuit on picked_up/arrived, the missing-order/missing-dispensary log+return, and the swallow-don't-rethrow contract under both `Error` and non-Error throws.

- **`NotificationsModule` wiring** (`apps/api/src/modules/notifications/notifications.module.ts` + `apps/api/src/app.module.ts`) — Composes the repositories (push_tokens, notifications, users, dispensaries, drivers, orders), the dedupe store (REDIS_CLIENT injection), the three real providers (APNs/Twilio/Resend constructed once at module init from `ConfigService`), the dispatcher, the order listener, and the push-tokens controller. APNs `production` flag derives from `NODE_ENV === 'production'` so dev hits the sandbox gateway. Twilio sender selection uses optional config — `TWILIO_MESSAGING_SERVICE_SID` if set, else `TWILIO_FROM_NUMBER` — matching the runtime shape in `packages/config/src/env.ts`. Provider constructors fail fast at boot if a required key is missing (`getOrThrow`). NotificationsModule sits last in the AppModule import list so its listener registers AFTER OrdersModule's EventEmitter binding, keeping Nest's bootstrap deterministic.

- **`TWILIO_MESSAGING_SERVICE_SID` + `TWILIO_FROM_NUMBER` env config** (`.env.example` + `packages/config/src/env.ts`) — Both optional. Messaging Service SID preferred (lets Twilio rotate senders out of a pool); from-number is the fallback for environments with a single dedicated long code. From-number validated against E.164 regex (`/^\+\d{8,15}$/u`). Distinct from the existing `TWILIO_PROXY_SERVICE_SID`, which drives the masked driver↔customer comms during a delivery (Phase 8) — different Twilio product, different SID format, intentionally not collapsed into one env var.

## Phase 11 — Metrc Traceability

_Status: complete (2026-05-19). Branch: `phase/11-metrc-traceability`. Commit range: `f9ec2cf..b7ba149` (10 commits)._

What landed:

- **`@dankdash/metrc` typed REST client** (`packages/metrc/`) — Pure HTTP layer (no NestJS, no DI graph). `MetrcClient` (`src/client.ts`) exposes the four endpoints the reporting + reconciliation paths need: `postSalesReceipt(input: PostSalesReceiptInput): Promise<void>` (200 with empty body — Metrc returns no receipt id on POST, the reconciler discovers it later), `listActiveReceipts(input: ListActiveReceiptsInput): Promise<MetrcReceipt[]>` (windowed by `lastModifiedStart..lastModifiedEnd` + `licenseNumber`), `getReceiptById(id, licenseNumber)`, and `voidReceipt(id, licenseNumber)`. The internal `HttpClient` (`src/http.ts`) is undici-backed with a shared `Dispatcher` configurable via `createUndiciDispatcher({maxConnections, keepAliveTimeoutMs})`. Per spec §7.3 we cap at 4 concurrent sockets across the integrator to stay inside the published rate window. `MetrcAuth` (`src/auth.ts`) handles the documented HTTP Basic auth (vendor key as username, per-licensee user key as password) — both arrive base64-encoded into a single `Authorization` header so a logged URL never leaks the user key as a query string. Every public method composes `Authorization` + `User-Agent: DankDash/<env>` + `Content-Type` + per-call retry policy. Zod-validated wire schemas in `src/schemas.ts` mean a Metrc shape regression is caught at the boundary, not in the consumer. 4 test files / 63 tests with **100% line + branch coverage** — covers happy path, 401/403/404/409/422/429/5xx response handling, retry-after honouring, the `vendor-api-key only` flavor where a license is global rather than per-receipt, and the schema rejection path (a malformed `Receipt.id` from upstream produces a typed `ExternalServiceError` rather than a runtime crash downstream).

- **`MetrcTransactionsRepository` + `next_retry_at` backoff column** (`packages/db/src/repositories/metrc-transactions.repo.ts`, migration `0008_metrc_next_retry_at`) — `metric_transactions` was already scaffolded in Phase 1; this phase makes it operational. New `next_retry_at timestamptz` column drives the per-row backoff ladder; `listReportedSince(since, until)` returns rows with `status ∈ ('reported','reconciled')` whose `reportedAt ∈ [since, until)` for the reconciliation sweep; `listDueForRetry(now, limit)` returns `pending|failed` rows whose `next_retry_at <= now` for the reporting cron; `markReported(id, reportedAt)` stamps `status='reported', reported_at=$1, next_retry_at=null`; `markReconciled(id, receiptId)` stamps the upstream Metrc receipt id once the reconciler matches a local row to an upstream receipt; `markFailedWithRetry(id, retryAt, error)` extends the ladder; `markFailedTerminal(id, error)` flips to `failed` and zeros the retry column. The split between `reported_at` (we POSTed; no receipt id yet) and `metrc_receipt_id` (reconciler resolved it) is the same split Metrc enforces server-side — POST returns 200 with empty body and the id surfaces on the next `GET /receipts/active` window. The fix commit `c7bd14f` is the post-mortem on an earlier draft that tried to write the receipt id at POST time and was blocked by that exact Metrc contract.

- **`OrdersRepository.findManyByIds(ids)` batched lookup** (`packages/db/src/repositories/orders.repo.ts`) — Single `WHERE id IN (...)` query used by the reconciliation cron to map a set of `metric_transactions.order_id` values back to their owning dispensary without an N+1. Empty input short-circuits before composing SQL (Drizzle's `inArray` on an empty list emits a degenerate `WHERE id IN ()` that some drivers reject outright and others read as "match nothing" — explicit contract beats either silent behavior).

- **`MetrcEnqueueListener` on `delivered`** (`apps/api/src/modules/compliance/metrc-enqueue.listener.ts` + `compliance.module.ts`) — `@OnEvent(ORDER_TRANSITIONED_EVENT, { suppressErrors: true })` subscriber that fires only on `toStatus === 'delivered'`, gated by the `ENABLE_METRC` env flag so dev environments don't accrue a pending backlog. Per delivery, queries `OrderItemsRepository.listForOrder(orderId)`, collects the `metrcPackageTag` of each item (filters `null` tags), and inserts a row into `metric_transactions` with `status='pending'`. Idempotency via `UNIQUE(order_id)` — duplicate `OrderTransitionedEvent` for the same order (e.g. event-bus at-least-once re-delivery) maps PG `23505` to a logged no-op. An empty tag array still inserts the row so the failure is visible at the worker (the row will fail terminal with "no package tags") rather than silent on the listener side — a beverage-only or otherwise-untagged delivery is a catalog-admission bug and we want it loud. Any other error is caught and logged at error-level so the listener never propagates back through the OrderTransitionService — the transition is already durable and the response path must not 500 because Metrc bookkeeping had a hiccup. 10 unit tests cover the happy path, the unique-violation skip, the missing-tag warning, the disabled-flag short-circuit, and the listener's swallow-don't-rethrow contract under both `Error` instances and non-Error throws.

- **`scheduleMetrcReportingJob` cron** (`apps/workers/src/jobs/metrc-reporting/`) — node-cron `*/1 * * * *` (every minute) America/Chicago. Per tick: `listDueForRetry(now, 50)` to pull a bounded batch of `pending|failed` rows whose `next_retry_at <= now`, group by `dispensaryId`, sequentially per dispensary decrypt the per-license credentials (`encryption.decrypt(dispensary.metricApiKeyEnc, ENCRYPTION_CONTEXT.DISPENSARY_METRC_API_KEY)`), and call `metric.postSalesReceipt` for each row. On 2xx: `markReported(id, now)`. On 4xx that the spec calls non-retryable (`400/401/403/404/422`): `markFailedTerminal` with the error envelope so ops sees the row in the failed bucket. On 429 or 5xx: `markFailedWithRetry(id, now + nextBackoff(attempt), error)` using the explicit ladder in `backoff.ts` (`30s, 2m, 10m, 1h, 6h, 24h` — capped at 24h per spec). The job's outer try/catch turns any per-row throw into a `markFailedWithRetry` so one bad row never poisons the rest of the dispensary's queue. `backoff.ts` is pure — 8 unit tests pin the exact schedule and the cap. The job file is 28 + 8 tests (36 total) covering the happy path, every classified-error arm, the per-dispensary credential decrypt failure (rewrapped as `ExternalServiceError('metrc', ...)` with `cause`), empty batch tick (zero-cost no-op), and the bounded fan-out — concurrent dispensaries are processed sequentially, concurrent rows for the same dispensary are processed sequentially against the per-license credential.

- **`scheduleMetrcReconciliationJob` cron** (`apps/workers/src/jobs/metrc-reconciliation/`) — node-cron `0 4 * * *` (daily 04:00) America/Chicago — late enough that the prior day's POSTed receipts are settled in Metrc's backend, early enough that the operator has a discrepancy report before opening hours. Per tick: `metricTransactions.listReportedSince(now - 7d, now)` returns every `reported|reconciled` row in the window across all dispensaries in a single query, then `orders.findManyByIds(ids)` batch-resolves each row's owning dispensary (avoids an N+1 over the same orders the reporter walked yesterday). Rows are grouped by `dispensaryId` in app code and the reconcile loop runs once per active dispensary. Per dispensary: `metric.listActiveReceipts({ licenseNumber, lastModifiedStart: now-7d ± 5min skew, lastModifiedEnd: now ± 5min skew })`, then for each local row, find the upstream receipt whose `transactions[].packageLabel` set covers the local row's `packageTags` set (⊇, not = — Metrc occasionally splits or merges lines during their own reconciliation, what matters is every package we sold appears on the receipt). On match: `markReconciled(row.id, String(receipt.id))`. Three discrepancy kinds get logged at error-level with structured `event: 'metrc.reconcile.discrepancy'` so the existing log-shipping alert pipeline can fire on them without an extra notifications hop: `missing_upstream` (local `reported` row >24h old with no covering upstream receipt — likely a Metrc-side drop), `unexpected_upstream` (an active receipt with no local row that covers it — likely a manual write in the Metrc UI we didn't author), `receipt_id_mismatch` (an already-`reconciled` row whose stored `metric_receipt_id` no longer matches the receipt currently covering it — upstream split or operator merge). The 5-minute window skew accommodates clock drift between our `reported_at` and Metrc's `LastModified`; matching is by tag set rather than timestamp so the skew is cosmetic to the algorithm. Per-dispensary credential decrypt failures rewrap to `ExternalServiceError('metrc', ...)` and the orchestrator's outer try/catch counts the dispensary as an error in the summary without aborting other dispensaries. 26 job tests (covering empty tick, single + cross-dispensary grouping, the three discrepancy kinds with edge cases, decrypt failure isolation, internal helpers via `__INTERNALS__`) + 3 scheduler tests (pins the `0 4 * * *` expression, the America/Chicago timezone, and the orchestration-failure swallow + log path so a bad tick never crashes the worker process).

- **Worker entrypoint wiring** (`apps/workers/src/main.ts`) — Both Metrc crons gated behind `env.ENABLE_METRC`; when off, the scheduler is never constructed so the per-row backoff ladder doesn't tick in a non-prod environment that can't actually reach the vendor. Constructs a dedicated `MetrcHttpClient` with the 4-connection undici dispatcher isolated from the Aeropay pool (different vendor, different keepalive characteristics). Both `metrcReportingTask` and `metrcReconciliationTask` join the SIGTERM shutdown chain so a Railway rolling restart drains the in-flight ticks cleanly. Shared DI graph between the two crons (same `dispensaries`, `metricTransactions`, `metrc`, `encryption`, `logger`) — the worker owns one DI graph and both jobs are wired off it.

## Phase 10 — Tracking & Geofencing

_Status: complete (2026-05-19). Branch: `phase/10-tracking-geofencing`. Commit range: `baf85f5..27f6834` (14 commits)._

What landed:

- **Location ingestion worker** (`apps/workers/src/jobs/location-ingest/`) — joins the `dankdash:realtime` Redis Stream as a consumer-group member (`realtime` → `location-ingest:<host>-<pid>`), filters to `driver:location` envelopes, batches up to `LOCATION_INGEST_BATCH_SIZE=50` items or `LOCATION_INGEST_BATCH_DELAY_MS=200ms`, and commits each batch in a single Postgres transaction: insert into `driver_location_history` (the weekly-partitioned table) + UPDATE `drivers.current_location` + XACK the stream entries. `XPENDING IDLE … XCLAIM` recovery picks up entries from a dead pod before the regular `XREADGROUP BLOCK`. The writer is split into `location-ingest.batcher.ts` (pure timing logic — 7 tests) + `location-ingest.writer.ts` (DB IO — 3 tests) + `location-ingest.consumer.ts` (Redis loop). The consumer fires an injected `onCommitted(item)` callback per item after the batch's XACK so downstream observers (geofence, ETA) get exactly-once invocations under the at-least-once stream contract.

- **`drivers.updateLocation` stale-write guard** — the location writer can race with itself under claim-recovery, so the UPDATE includes a `WHERE current_location_updated_at IS NULL OR current_location_updated_at < $stamp` predicate. An earlier ping that arrives second silently no-ops instead of overwriting the fresher one. The first cut used a raw `sql\`... < ${stamp}\`` template that lost the column's timestamptz type on its way to postgres.js (`ERR_INVALID_ARG_TYPE`); routed through drizzle's typed `lt()`+`isNull()`+`or()` in the fix.

- **Geofence arrival observer** (`apps/workers/src/jobs/location-ingest/geofence.*`) — pure haversine math in `geofence.service.ts` (23 tests: meters, km, equator, poles, antipode, the 50m threshold, the dropoff-snapshot parser) plus the observer that wraps it (`geofence.observer.ts`, 12 tests). Per ping: cheap pre-check on order status (only `en_route_dropoff` is transitionable); if inside the 50m threshold, calls `OrdersRepository.applyTransition('DRIVER_ARRIVED')` which row-locks under `SELECT … FOR UPDATE` and re-runs the state machine — concurrent pings throw `ORDER_INVALID_TRANSITION` from the resolver and we swallow them. Non-OrderError exceptions rethrow so the consumer's outer warn-log captures them. The threshold and dropoff-point extraction are exported as pure functions so they're testable without a DB.

- **`@dankdash/eta` cache-fronted ETA service** (`packages/eta/`) — `EtaService.computeEta(from, to)` quantizes both endpoints to a grid (`DEFAULT_GRID_PRECISION_DEGREES`, ~110m at 45°N), looks up `eta:v1:<from>:<to>` in Redis with a sub-second timeout, calls Mapbox Directions API on miss with a 60s SETEX cache, and falls back to `haversineMeters × 0.8` (spec §10.3) on Mapbox failure without caching the fallback. The `EtaResult` carries a `source: 'cache' | 'mapbox' | 'fallback'` discriminant so observability can split cache-hit ratio from real-call latency. The `MapboxClient` adds an `ExternalServiceError`-wrapping retry-aware Directions caller. Tests cover grid quantization, distance math, cache hit/miss/poisoning, Mapbox 5xx fallback, and the 0.8 multiplier boundary.

- **ETA observer + realtime `customer:eta_updated` event** — every committed location ping for a driver with an active order publishes a `customer:eta_updated` envelope onto `dankdash:realtime`. The realtime service routes it to the `customer:{userId}` room so the consumer iOS app can paint a fresh ETA without polling. The event type is added to the discriminated `RealtimeEvent` union in `@dankdash/realtime-events` (zod schema enforces `{orderId, etaSeconds, distanceMeters, source}` at the wire). 13 observer tests cover the publish path including null-active-order skip, ExternalServiceError propagation, and idempotency under repeat pings.

- **Partition lifecycle cron** (`apps/workers/src/jobs/partition-management/`) — a node-cron `30 2 * * 0` America/Chicago job that runs three phases per Sunday tick:
  1. **Drop orphans**: any `<parent>_YYYY_wNN` table not in `pg_inherits` (i.e. leftover from a prior crashed run) gets dropped.
  2. **Ensure future**: creates 4 ISO-week-numbered partitions ahead of `now` via the `dankdash_create_week_partition(parent, week_start)` SQL helper baked into the bootstrap migration. Idempotent — calling for an existing week is a no-op.
  3. **Archive + drop expired**: for each partition whose `range_end < now - retentionDays` (default 90), streams rows to a Parquet archive in R2, then `DETACH PARTITION` + `DROP TABLE`. Per-partition try/catch isolates failures so a single bad week doesn't poison the rest.

  The `PartitionLifecycleService` depends on an injected `PartitionArchiver` interface; tests use a fake archiver and production wires `ParquetPartitionArchiver`. 20 service tests cover orchestration, per-step failure isolation, ISO-week-year boundary (2024-12-30 → `2025_w01`, 2025-12-29 → `2026_w01`), and the retention/lookahead overrides.

- **`PartitionsRepository`** (`packages/db/src/repositories/partitions.repo.ts`) — typed surface over the partition catalog. `createWeekPartition` (forwards to the SQL helper), `listWeekPartitions` (parses `pg_get_expr(relpartbound, oid)` text back into typed Date ranges), `detachPartition` + `dropTable` (explicit two-step rather than DETACH CONCURRENTLY because the 02:30 Sunday window absorbs the brief AccessExclusiveLock), `listDetachedOrphans` (matches `<parent>_YYYY_wNN` not in `pg_inherits`), and `streamPartitionRows` (async generator with BigInt-cursor keyset pagination, unpacks PostGIS geography to lat/lng doubles via `ST_Y`/`ST_X` so the archiver doesn't need geo plumbing). Every identifier crosses a `SAFE_IDENTIFIER` regex before composing SQL — Postgres rejects bind parameters for object identifiers, so the guard is the only line. 7 integration tests against the testcontainers Postgres exercise the full create → list → stream → detach → orphan → drop cycle plus safety rejects.

- **`ParquetPartitionArchiver`** (`apps/workers/src/jobs/partition-management/parquet-archiver.ts`) — end-to-end streaming pipeline: `PartitionsRepository.streamPartitionRows` → `@dsnp/parquetjs` ParquetWriter (GZIP per column) → `ByteCountingPassThrough` Transform (gives the encoded size for the summary without a HEAD round-trip) → R2 `@aws-sdk/lib-storage` Upload (multipart 5MB chunks). Schema choices: `id` is UTF8 not INT64 (bigserial values cross JS Number safe range and parquetjs's INT64 handling has BigInt/number ambiguity across Athena/DuckDB versions; UTF8 round-trips losslessly and consumers cast back); `recorded_at` is TIMESTAMP_MILLIS (JS Date.getTime() lands directly); geography unpacked to lat/lng doubles upstream (Parquet has no geometry type). GZIP rather than SNAPPY — archive bytes are written once and read rarely (regulator export), so we optimize storage density over decode speed. Object key is `archives/driver_location_history/<partition>.parquet` — content-addressed by partition name, so a re-run after a mid-pipeline failure overwrites identical bytes and gets idempotency without bookkeeping. 6 tests including a real round-trip via `ParquetReader.openBuffer` that catches schema/shape regressions mock-only tests would miss.

- **`@dankdash/storage` `putObject` + `putObjectStream`** — adds two new methods to the existing `R2Storage` surface. The streamed variant wraps `@aws-sdk/lib-storage` Upload (4 parts in flight, 5MB each) and absorbs the multipart-completion plumbing. Bumps `@aws-sdk/client-s3` from `^3.700.0` to `^3.1050.0` to satisfy lib-storage's peer range. Both surfaces share `assertKey` and wrap unexpected errors in `ExternalServiceError`.

- **Worker entrypoint wiring** (`apps/workers/src/main.ts`) — constructs `PartitionsRepository`, a dedicated `R2Storage` client (separate from the API's because the worker owns its own DI graph), `ParquetPartitionArchiver`, and registers `schedulePartitionManagementJob`. Adds `partitionTask.stop()` to the graceful-shutdown chain.

## Phase 9 — Realtime Service (Socket.io)

_Status: complete (2026-05-19). Branch: `phase/09-realtime`._

What landed:

- **`@dankdash/realtime-events` package** (`packages/realtime-events/`) — single source of truth for the wire format on the shared `dankdash:realtime` Redis Stream. Exports a `RealtimeEnvelope` zod schema (`{id, emittedAt, source, event}`) wrapping a discriminated `RealtimeEvent` union (`order:status_changed`, `order:driver_assigned`, `driver:location`, `vendor:notification`). `publishRealtimeEvent` XADDs an `envelope`-keyed JSON field; `decodeStreamEntry` validates back to typed shape so the consumer can switch on `event.type` exhaustively. Producers (API, workers) and the realtime service all import from this barrel — adding a new variant breaks `tsc` everywhere until handled.
- **Standalone Socket.io service** (`apps/realtime/`) — Express + Socket.io on a dedicated port, deployable independently from the API on Railway. The `@socket.io/redis-adapter` (`dankdash:io:` key prefix) lets multiple pods see each other's rooms behind the TCP proxy. JWT auth is RS256 against the same public key the API issues: `RealtimeJwtVerifier` decodes a base64 PEM once at boot, verifies iss/aud/sub/role/sid, and stashes `{token, claims}` on `socket.data`. The `MembershipRepository` interface is dual-implemented — `DrizzleMembershipRepository` for prod (`isStaffOfDispensary`, `findDriverIdForUser`), `InMemoryMembershipRepository` for the test harness.
- **Three namespaces, three audience scopes** (`apps/realtime/src/io/namespaces/`). `/customer` auto-joins `customer:{userId}` plus `order:{orderId}` when an active order is named on the handshake. `/vendor` requires the handshake to name a `dispensaryId`, runs the staff lookup, then joins `dispensary:{id}` — without that check every staff token would broadcast across all dispensaries. `/driver` resolves the driver record off the JWT sub (or the explicit `driverId` handshake field), joins `driver:{driverId}`, and exposes two client-to-server events: `driver:heartbeat` (replies with an ack carrying a server timestamp so the client can detect a one-way break) and `driver:location:update` (zod-validated `{lat, lng, accuracyMeters?, speedMps?, headingDeg?, batteryPct?, orderId?, customerId?}`, in-memory token-bucket rate-limited via `DRIVER_LOCATION_BURST` + `DRIVER_LOCATION_RATE_PER_SECOND` env knobs, emits `driver:location:rate_limited` on starvation). On accept it publishes a `driver:location` envelope to the realtime stream — the only place an unprivileged client writes into the bus.
- **`StreamConsumer`** (`apps/realtime/src/streams/`) — one consumer group (`realtime` by default), one consumer name per pod (`hostname-pid`). The loop is `XPENDING IDLE … XCLAIM` recovery first (entries from a dead pod get picked up by survivors), then `XREADGROUP BLOCK 5000` for new entries; each entry decodes through `decodeStreamEntry`, routes via the pure `routeEnvelope` switch to `{namespace, room, eventName, payload}` broadcasts, then XACKs. Decode failures are ACKed (poison entries get logged once, not retried forever). `stop()` flips a running flag and awaits the in-flight BLOCK so SIGTERM redeploys are clean.
- **Two Redis connections, not one** (`apps/realtime/src/server.ts`) — the @socket.io/redis-adapter installs a `dankdash:io:` key prefix on the pubClient that would silently shift our application stream key, so the service builds a separate `streamClient = new Redis(REDIS_URL)` for XADD/XREADGROUP. And because ioredis serializes commands per connection, `XREADGROUP BLOCK 5000` on a shared connection would deadlock the producer's XADD for 5s — `consumerClient = streamClient.duplicate()` gives the blocking consumer its own connection. Both gotchas root-caused live during Phase 9.5 (initially every `driver:location:update` test hung silently); the comments in `server.ts` document the invariants so the next person doesn't re-derive them under fire.
- **Composition root + bootstrap** — `buildServer({env, logger, pool?, membership?})` is the single factory: no global singletons, no import-time side effects, returns `{app, httpServer, io, listen, close}`. Tests inject an in-memory membership and skip Postgres entirely. `close()` shuts the consumer first, then `io.close()` (which also tears down the engine.io-attached httpServer — the explicit `httpServer.close()` is guarded behind `httpServer.listening` to avoid ERR_SERVER_NOT_RUNNING). `main.ts` is the process entrypoint — loads env, builds pino, constructs the `@dankdash/db` pool, calls `buildServer`, and binds SIGTERM/SIGINT to `server.close()` for graceful redeploys.
- **`/healthz`** returns 200 once the Redis stream client reports `ready` — Railway's load balancer gates pod traffic on it.
- **22 integration tests against a Redis 7 testcontainer** (`apps/realtime/test/`). `global-setup.ts` boots one container for the whole run and exports `REDIS_TEST_URL`. `harness.ts` exposes `createTestHarness()` per `beforeEach`: mints an RSA keypair, threads the public key into `loadRealtimeEnv`, picks a unique consumer-group name (`realtime-test-<rand>`) so concurrent specs don't steal each other's XREADGROUP entries, returns `signToken`, `connect(namespace, opts)`, `publishEnvelope`, `close`. Coverage: `auth.test.ts` (13 — valid tokens across all three roles, missing/expired/wrong-issuer/role-mismatch rejection, staff/driver membership enforcement, stream→room broadcasts), `rate-limit.test.ts` (5 — happy path, token-bucket starvation, zod validation, heartbeat ack, direct XLEN/XREVRANGE wire-format inspection), `streams.test.ts` (4 — order-status routing, vendor-notification routing, malformed envelope swallowed).

## Phase 8 — Dispatch & Driver Foundation

_Status: complete (2026-05-19). Branch: `phase/08-dispatch-driver`. Commit range: `8d04b5b..e7c7184` (12 commits). PR: [#9](https://github.com/jwillz7667/DankDash/pull/9)._

What landed:

- **`dispatch_failed` terminal state** — adds the operational-failure sink for "no driver in the polygon picked up within the offer window" so an order never sits in `awaiting_driver` indefinitely. Migration `0006_dispatch_failed` adds the enum value; migration `0007_dispatch_failed_column` adds the `dispatch_failed_at` timestamptz column and widens `orders_active_idx` (the partial index over hot orders) to exclude the new terminal value. The split exists because Postgres rejects USE of a newly-added enum value in the same transaction that ADDed it (error 55P04 — `new enum values must be committed before they can be used`); 0007 references the value once 0006 has committed. The 0006 down-migration is a documented no-op — Postgres has no `ALTER TYPE … DROP VALUE`. `STATUS_TIMESTAMP_COLUMN` in `packages/db/src/repositories/orders.repo.ts` learns the new state so the transition service stamps `dispatch_failed_at` automatically. `@dankdash/orders` adds the state to `ORDER_STATES` (between `awaiting_driver` and `driver_assigned`) and to `TERMINAL_ORDER_STATES`; `events.ts` adds `DISPATCH_FAILED`; the XState machine accepts `awaiting_driver → dispatch_failed` from the system actor. The integration mirror test (`packages/db/test/integration/order-transitions.test.ts`) keeps the enum + state tuple + iOS arm in lockstep.
- **`@dankdash/dispatch` package** (`packages/dispatch/`) — pure scoring + attempt orchestration with no NestJS / no Drizzle / no HTTP. `scoring.ts` exposes `scoreDriver(candidate, dispensary, now)` returning a non-negative integer composite of three weighted sub-scores: `distanceScore` (1 − `min(distanceMeters, RADIUS_METERS) / RADIUS_METERS`) × `DISTANCE_WEIGHT (0.6)`, `ratingScore` (driver rating normalized to `[0,1]` over `RATING_FLOOR=3.0..RATING_CEILING=5.0`) × `RATING_WEIGHT (0.25)`, and `recencyScore` (older `last_completed_delivery_at` ranks higher to avoid starving newer drivers; saturates at `RECENCY_HORIZON_SECONDS=21600` = 6 hours) × `RECENCY_WEIGHT (0.15)`. Weights sum to 1.0 and are exported via `SCORING_WEIGHTS` for test reflection. `rankCandidates` sorts deterministically (score DESC, then driver `id` ASC for ties) so two identically-scored candidates never flip between worker ticks. `attempt.ts` exposes `attemptDispatch(orderId, port)` which loads candidates from the `DispatchPort` interface, scores + ranks, iterates: for each candidate creates an offer (`port.createOffer({ orderId, driverId, expiresAt: now + OFFER_TTL_SECONDS })`), awaits a port-side resolution, advances if `declined`/`expired`/`replaced`, returns `{ outcome: 'accepted', driverId, offerId }` on accept. If the candidate list is exhausted within the global window (`ATTEMPT_WINDOW_SECONDS=180`), returns `{ outcome: 'dispatch_failed', reason: 'no_candidates' | 'all_declined' | 'all_expired' }` so the worker can transition the order to `dispatch_failed` without making the decision itself. `DispatchPort` declares the surface the worker injects: `findCandidates`, `createOffer`, `awaitResolution(offerId, deadline)`, `markFailed`. The package is `vitest`-tested with hand-rolled fake ports — 27 unit tests, 100% line + branch coverage.
- **Dispatch worker jobs** (`apps/workers/src/jobs/dispatch/`) — two BullMQ jobs that drive the dispatch loop without coupling the API process to the long-poll wait. `dispatch.job.ts` subscribes to `OrderTransitionedEvent` for the `awaiting_driver` arrival, instantiates a real `DispatchPort` adapter wrapping `OrdersRepository.findDispatchCandidatesNearDispensary` + `DispatchOffersRepository.createForDriver` + the offer-resolution Redis pub/sub channel, calls `attemptDispatch`, and on `accepted` lets the offer-accept HTTP route flip the order on the driver's confirm; on `dispatch_failed`, it calls `OrdersRepository.applyTransition` with the `DISPATCH_FAILED` event so the transition is atomic with the audit insert. `offer-expiry.job.ts` is a 1-second tick that finds offers past `expires_at` in state `offered` and atomically flips them to `expired` via `DispatchOffersRepository.expireOtherActiveForOrder`-style `UPDATE … WHERE id = $1 AND status = 'offered' RETURNING …`, then publishes the resolution event back to whichever attempt loop is parked on it. Both jobs use the BullMQ rate-limiter to cap concurrent ticks per Redis instance (`maxJobsPerSecond=5`) and have unit tests against a fake repo + a fake clock (19 tests across both jobs).
- **Driver repository extensions** (`packages/db/src/repositories/`):
  - `OrdersRepository.findDispatchCandidatesNearDispensary(dispensaryId, opts)` — single Postgres+PostGIS query returning a `DispatchCandidateRow[]` with `id`, `latitude`, `longitude`, `rating`, `distanceMeters`, `lastCompletedDeliveryAt`. Uses `ST_DWithin(driver.current_location::geography, dispensary.location::geography, $radiusMeters)` to push the radius check into the spatial index rather than scoring every online driver. Filters: `current_status = 'online'`, `current_order_id IS NULL`, `current_location IS NOT NULL`. The radius defaults to `DISPATCH_DEFAULT_RADIUS_METERS=16093` (10 miles per spec).
  - `OrdersRepository.listInStatus(status, opts)` — pagination by `status_changed_at DESC` for the dispatch worker's catch-up sweep on startup (so a worker restart doesn't strand orders parked at `ready_for_pickup`).

  - `DispatchOffersRepository.expireOtherActiveForOrder(orderId, exceptOfferId, tx)` — when one offer is accepted, expires every other `offered` row for the same order in one UPDATE so the offer table is the single source of truth on "what's still live".

The guard's own tests cover the missing-row, suspended-driver, and admin-impersonation paths (admin override is explicit, not implicit).
- **Driver admin endpoints** (`apps/api/src/modules/drivers/admin/`) — `POST /v1/admin/drivers` (admin approves a driver application — body contains user id, license number, vehicle make/model/plate, insurance carrier + policy number + expiry date; pgcrypto-encrypted at the column level per the data-classification rules), `PATCH /v1/admin/drivers/:id/status { status: 'active' | 'suspended' }` (suspending an active driver also cancels any in-flight offers and flips their `current_status` to `offline` in the same tx). Both routes behind `JwtAuthGuard + RateLimitGuard + AdminGuard`. The service returns the projected driver via `projectDriver()` which omits restricted fields by default and only includes the license-number suffix (last 4) so the admin UI can render a confirmation without the full plaintext.
- **Shift management endpoints** (`apps/api/src/modules/drivers/shift/`) — `POST /v1/driver/shift/start { startingLocation: { latitude, longitude } }`, `POST /v1/driver/shift/end`, `POST /v1/driver/status { status: 'online' | 'offline' | 'on_break' }`. Start inserts a row in `driver_shifts` and flips the driver's `current_status` to `online` + `current_location` to the body coords. End closes the open shift (`ended_at = now()`), refuses if the driver has an active order (`DRIVER_SHIFT_HAS_ACTIVE_ORDER`), and flips `current_status` to `offline`. The status endpoint is the per-shift "I'm taking a break" toggle. All three routes go through `DriverContextGuard` and `@CurrentDriver()`; tests cover the missing-shift cases (end-with-no-open-shift surfaces 409 with `DRIVER_SHIFT_NOT_OPEN`).

Accept runs inside one transaction: `findByIdForUpdate(offerId)` to lock the offer row, asserts `status === 'offered'` and `expires_at > now()` (otherwise `DRIVER_OFFER_NOT_AVAILABLE`), asserts the offer's driver matches the calling driver (otherwise `DRIVER_OFFER_NOT_YOURS`), composes `OrderTransitionService.transitionWithinTx(tx, orderId, { event: 'DRIVER_ASSIGNED', actor: { role: 'system' }, patch: { driverId } })` so the order's `status` flips to `driver_assigned` and the audit insert lands in the same tx as the offer flip, `expireOtherActiveForOrder(orderId, offerId)` cancels sibling offers, UPDATEs `drivers.current_order_id` + `drivers.current_status='en_route_pickup'`, then commits. Decline flips just the offer to `declined` with the reason and emits `OfferDeclinedEvent` so the worker can re-attempt the next candidate. Tests pin the cross-tenant guards (a driver presenting another driver's offer id gets a 403 surfaced as a 404 — no info leak), the expired-offer path, and the race where two drivers POST accept simultaneously (the second loses the `SELECT … FOR UPDATE` race and surfaces `DRIVER_OFFER_NOT_AVAILABLE`).

Public surface is unchanged for the existing chokepoint callers; `transition()` now wraps `transitionWithinTx` inside `applyTransition`'s tx. Direct UPDATEs to `orders.status` or INSERTs into `order_events` / `order_status_history` remain forbidden — the exhaustive `AUTH_BY_EVENT` matrix runs inside `transitionWithinTx` so composed transitions get the same defense-in-depth as the top-level chokepoint.
- **Driver self-service app endpoints** (`apps/api/src/modules/drivers/app/`) — three read-only routes for the driver iOS app:
  - `GET /v1/driver/current-route` (rate cap 120 req/min — polled hot while driving). Joins `drivers.current_order_id` (set by offer-accept, cleared by delivery completion) → `orders` → `dispensaries`, projects the pickup address from the dispensary, and the dropoff from `orders.delivery_address_snapshot` via Zod `safeParse`. ## Phase 7 — Order Lifecycle & State Machine

_Status: complete (2026-05-19). Branch: `phase/07-order-lifecycle`. Commit range: `dfc38dc..0f848ca` (10 commits)._

What landed:

- **`@dankdash/orders` package** (`packages/orders/`) — pure state-machine package with no NestJS / no Drizzle / no HTTP. `states.ts` declares `ORDER_STATES` as a `readonly` tuple (19 states) and `TERMINAL_ORDER_STATES` (6: `payment_failed`, `rejected`, `delivered`, `returned_to_store`, `canceled`, `disputed`) — the tuple mirrors the `order_status` Postgres enum and the iOS `OrderStatus` enum, and an integration mirror test in `apps/api/test/integration/orders/` will catch drift. `order-machine.ts` builds the XState v5 machine via `setup({ types: { events: {} as OrderMachineEvent } }).createMachine(...)` so every event payload is typed at the call site. The machine is intentionally a pure transition table — no actions, no entry/exit side effects, no actors — so the `OrderTransitionService` can consult it inside a DB transaction without paying actor-spawn overhead. Public `nextOrderState(from, event)` runs a single-shot actor and returns the next state (or throws `OrderError.invalidTransition` with stable code `ORDER_INVALID_TRANSITION`). `isTerminalOrderState` short-circuits any event against a closed-final state before the machine spins up. `errors.ts` defines the `OrderError` discriminated subtype of `DomainError` with codes `ORDER_NOT_FOUND` (404), `ORDER_INVALID_TRANSITION` (422), `ORDER_ACTOR_FORBIDDEN` (403), `ORDER_TERMINAL_STATE` (422), and `ORDER_INVARIANT_BROKEN` (500). `events.ts` declares `OrderEventType` (a closed string union covering every edge — `VENDOR_ACCEPT`, `CUSTOMER_CANCEL`, `DRIVER_PICKED_UP`, `ID_SCAN_PASSED`, …) so adding an event there forces a matching case in the authz matrix and the per-state timestamp lookup.
- **Migration 0005 — `order_status_history` table + per-state timestamps** (`packages/db/src/migrations/0005_order_status_history.sql` + `.down.sql`) — adds 12 nullable `timestamptz` columns to `orders` (`payment_failed_at`, `rejected_at`, `prepping_at`, `awaiting_driver_at`, `driver_assigned_at`, `en_route_pickup_at`, `en_route_dropoff_at`, `arrived_at_dropoff_at`, `id_scan_pending_at`, `returned_to_store_at`, `disputed_at`, `rated_at`) so the transition service can stamp the matching column in the same UPDATE that flips `status` — O(1) "how long has this order been waiting on the driver" queries without joining the history table. Existing `placed_at`, `accepted_at`, `prepared_at`, `picked_up_at`, `delivered_at`, `canceled_at` keep their historical names. The new `order_status_history` table is partitioned monthly on `changed_at` (PK = `(id, changed_at)` per Postgres' partitioned-table rule), 13 partitions bootstrapped (current month + 12 forward), and the `dankdash_block_mutation` trigger from migration 0000 is re-applied so UPDATE/DELETE are rejected at the DB tier even from the app role. The monthly partition rollover function is `CREATE OR REPLACE`'d to include `order_status_history` alongside `order_events`, `notifications`, `audit_log`, and `driver_location_history` — same single-source list, no fan-out across crons. The down migration drops the history table, restores the original rollover function, and removes the 12 columns.
- **`OrdersRepository.applyTransition(orderId, resolver)`** (`packages/db/src/repositories/orders.repo.ts`) — the new authoritative entry point for every status change. Opens a transaction, runs `SELECT … FOR UPDATE` on the order row, hands the locked snapshot to the caller's resolver, and the resolver returns a `TransitionDecision { toStatus, eventType, actorUserId?, actorRole?, payload?, reason?, patch? }`. With that decision the repo UPDATEs `orders.status` + `status_changed_at` + the per-state timestamp column derived from `STATUS_TIMESTAMP_COLUMN` (the lookup ensures the column name is never hand-rolled at the call site), spreads any `patch` fields (driverId, canceledBy, cancelReason, deliveryIdScan\*), INSERTs the immutable `order_events` row, INSERTs the typed `order_status_history` row — all inside the same tx. Throws `NotFoundError` if the row vanishes; `RepositoryError` on impossible "UPDATE … RETURNING returned no row" cases. The legacy `transitionStatus` method is marked `@deprecated` and remaining call sites have been migrated.
- **`OrderTransitionService`** (`apps/api/src/modules/orders/order-transition.service.ts`) — the **single chokepoint** for every UPDATE to `orders.status` and every INSERT into `order_events` / `order_status_history`. No other code path in the API is allowed to mutate order state; this is documented in the file header. The service runs an exhaustive `AUTH_BY_EVENT: Readonly<Record<OrderEventType, AuthFn>>` matrix where `Record<OrderEventType, …>` forces a compile-time match for every event in `@dankdash/orders/events.ts` — adding an event without an authz rule breaks `tsc`. Predicates: `vendorOwnsOrder` (vendor matches `dispensary_id` from the locked row + admin override), `customerOwnsOrder`, `assignedDriver` (driver row.driver_id matches the principal user_id), `allowSystemOrAdmin` (webhook + cron paths). Authorization runs **inside the lock** so a concurrent VENDOR_ACCEPT race sees the post-write status and bails with `ORDER_INVALID_TRANSITION` rather than silently double-writing. Defense-in-depth: a terminal state (except `delivered` → `disputed`) is rejected with `ORDER_TERMINAL_STATE` before the machine is consulted. The `fromStatus` is captured into a holder object (`{ value?: OrderState }`) so the post-commit unwrap is an explicit `OrderError.invariantBroken` throw rather than a non-null assertion — the assertion would have been "unreachable" but the holder pattern makes the unreachability the type checker's concern, not a TS-eslint suppression. Post-commit, the service emits `OrderTransitionedEvent` via EventEmitter2; if a subscriber throws, it is logged and swallowed — the DB has already committed, the caller's mutation is durable, and a downstream failure must not surface a 500 that suggests rollback.
- **`OrderTransitionedEvent` + EventEmitter2 plumbing** (`apps/api/src/modules/orders/order-transition.events.ts`) — typed event class carrying `{ orderId, fromStatus, toStatus, event, actor: { userId?, role, dispensaryId? }, occurredAt }`. `ORDER_TRANSITIONED_EVENT` constant is the emit key. `EventEmitterModule.forRoot({ wildcard: true, delimiter: '.', maxListeners: 20, verboseMemoryLeak: false })` is registered globally in `AppModule` so every feature module can publish/subscribe without redundant per-module imports. Subscribers consume the typed event class, not the raw payload — so a future schema change surfaces as a TS error in every subscriber. No subscribers wired in this phase (those land with realtime/notifications/dispatch in Phases 8–10); the publish surface is what we're standing up here.
- **`VendorOrdersController`** (`apps/api/src/modules/orders/vendor-orders.controller.ts`) — `/v1/vendor/orders/:id/{accept,reject,prepped,ready,handoff}` behind `JwtAuthGuard` → `RateLimitGuard` → `VendorContextGuard` (X-Dispensary-Id header). - **MFA TOTP** — speakeasy-backed `MfaService` with 30-second windows ±1 step. Secrets are AES-256-GCM-encrypted at the column layer (envelope-wrapped DEK under `COLUMN_ENC_MASTER_KEY`) so DB-only access never yields plaintext secrets. `disableMfa` requires a current code so a stolen access token alone cannot strip the second factor.
- **Persona KYC** — `PersonaService.createInquiry` posts to `withpersona.com/api/v1/inquiries`; `handleWebhook` verifies the `Persona-Signature` header (`t=…,v1=…`) with HMAC-SHA256 + a ±300s replay window + constant-time comparison, parses the JSON:API envelope through Zod, and dispatches `inquiry.completed`/`failed`/`expired` to discriminated-union outcomes. Age gate (MN Stat. § 342.46 minimum 21) is enforced at the completion outcome — under-21 raises `KYC_AGE_UNDER_MINIMUM` rather than silently completing.
- **Identity service** — `IdentityService.getMe`/`updateMe`/`startKyc`/`applyKycOutcome` orchestrate the user-facing flows. `applyKycOutcome` is the only place that flips `users.status=active` + stamps `kyc_verified_at` + persists the Persona-verified DOB over the client-typed value.
- **Auth + Identity controllers (9 endpoints)** — `/v1/auth/{register,login,refresh,logout,mfa/setup,mfa/confirm,mfa/verify,mfa/disable}` + `/v1/me` (GET + PATCH) + `/v1/identity/kyc/start` + `/v1/identity/kyc/webhook`. Public/protected boundary enforced by a global `JwtAuthGuard` (deny-by-default) with `@Public` as the opt-out, and `RolesGuard` (allow-list) for role-restricted routes — surfacing `@Public + @Roles` controller mistakes as 403 rather than silently letting them through.
- **Zod DTOs (`nestjs-zod`)** — single `.strict()` schema per endpoint serves the request DTO, response type, and (future) OpenAPI generation. Login uses a discriminated-union response (`status: 'authenticated' | 'mfa_required'`) so the two-step MFA flow is type-safe end-to-end. The KYC webhook DTO is documentation-only; the controller uses `@RawBody()` because re-serializing a Zod-parsed object would invalidate the HMAC.
- **Column encryption** — `EncryptionService` provides AES-256-GCM envelope encryption with a 32-byte DEK per encryption call wrapped by `COLUMN_ENC_MASTER_KEY` (loaded from Railway secrets); ciphertext layout is `version(1) | wrapped_dek(60) | iv(12) | tag(16) | payload(N)` so future master-key rotation is a wrap-only re-encryption pass. Used by MfaService for `mfa_secret_enc`.
- **Rate limiting** — `RateLimitGuard` keyed on a sha256-truncated tracker (`ip`, `user`, `email-from-body`, `refresh-from-body`) backed by Redis `INCR + PEXPIRE NX + PTTL` pipeline (fixed window, one round trip). Multi-tracker `@RateLimit` decorator: `/auth/login` is `5/min per-IP AND 10/hour per-email` simultaneously. `MemoryRateLimitStore` provides a Redis-free fallback for `NODE_ENV=test`. RateLimitError details carry `retryAfterSeconds` for client back-off.
- **234 unit tests, 16 files** — covers password (20), jwt (9), refresh-token (22), mfa (20), persona (32), encryption helper, auth.service (17), identity.service (15), auth.controller (9), identity.controller (7), jwt-auth.guard (7), roles.guard (6), rate-limit.guard (12), zod-validation.pipe (5), DTO schemas (auth 34 + identity 13), bootstrap smoke (6). All instantiate components directly with hand-rolled fakes — Nest container is only spun in the bootstrap smoke suite.
- **Eight new error classes in `@dankdash/types`** — `PasswordError`, `EncryptionError`, `ConfigError`, `KycError`, plus expansions to `AuthError` (`UNAUTHENTICATED`, `MFA_CODE_INVALID`). Every controller-reachable failure now has a stable code; the global filter maps these to HTTP statuses without leaking internals.
