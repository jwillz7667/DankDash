Composite uniqueness is on `(provider, session_id)` — distinct providers (Persona for signup, Veriff for handoff) may share session-id formats, so cross-provider uniqueness is not imposed.
- **`OrdersRepository.transitionStatus` ID-scan gate** — when `toStatus === 'delivered'`, the `FOR UPDATE` pre-read asserts `current.deliveryIdScanPassed === true` and otherwise throws `ConflictError('COMPLIANCE_ID_SCAN_REQUIRED')`. Sits at the same layer as the FROM-state guard so a future caller bypassing `DriverOrdersService` still cannot reach delivered without a Veriff-approved scan recorded on the row. The same `FOR UPDATE` lock serializes two-driver races on the same order — no extra round trip, no extra lock. Per ADR 0007 Decision 1.
- **Veriff client (`apps/api/src/modules/identity-verification/veriff.client.ts`)** — HMAC-SHA256 signed requests against `https://stationapi.veriff.com` (sandbox + production share the host, key routing decides). `createSession({ orderId, customerName, dateOfBirth })` posts a Veriff session and returns `{ verificationId, sessionUrl, sessionToken }`. `fetchDecision(verificationId)` polls `/v1/sessions/:id/decision` until `approved | declined | resubmission_requested | review`. `handleWebhook(rawBody, signatureHeader)` recomputes `HMAC-SHA256(secret, rawBody)`, constant-time-compares against `X-HMAC-SIGNATURE`, parses the JSON envelope through Zod, and returns a typed `VeriffWebhookDecision` discriminated union. All wire surface mocked in `veriff.client.test.ts` against `nock`; signature roundtrip + payload shape + replay window covered.
- **`POST /v1/webhooks/veriff`** — Fastify `@RawBody()` controller — re-serializing a Zod-parsed body would invalidate the HMAC, so the controller hands the raw `Buffer` directly to `VeriffClient.handleWebhook()`. On `approved`, writes `orders.delivery_id_scan_passed = true`, `delivery_id_scan_ref = verification.id`, `delivery_id_scan_at = now()` + inserts `age_verifications (provider, provider_session_id, …)`. ON CONFLICT path on the unique index turns a Veriff retry into a no-op. Invalid signature → 401; missing header → 400; both responses are intentional 4xx so Veriff stops retrying the bad payload.

- **`ParquetPartitionArchiver`** (`apps/workers/src/jobs/partition-management/parquet-archiver.ts`) — end-to-end streaming pipeline: `PartitionsRepository.streamPartitionRows` → `@dsnp/parquetjs` ParquetWriter (GZIP per column) → `ByteCountingPassThrough` Transform (gives the encoded size for the summary without a HEAD round-trip) → R2 `@aws-sdk/lib-storage` Upload (multipart 5MB chunks). Schema choices: `id` is UTF8 not INT64 (bigserial values cross JS Number safe range and parquetjs's INT64 handling has BigInt/number ambiguity across Athena/DuckDB versions; UTF8 round-trips losslessly and consumers cast back); `recorded_at` is TIMESTAMP_MILLIS (JS Date.getTime() lands directly); geography unpacked to lat/lng doubles upstream (Parquet has no geometry type). GZIP rather than SNAPPY — archive bytes are written once and read rarely (regulator export), so we optimize storage density over decode speed. Object key is `archives/driver_location_history/<partition>.parquet` — content-addressed by partition name, so a re-run after a mid-pipeline failure overwrites identical bytes and gets idempotency without bookkeeping. 6 tests including a real round-trip via `ParquetReader.openBuffer` that catches schema/shape regressions mock-only tests would miss.

The streamed variant wraps `@aws-sdk/lib-storage` Upload (4 parts in flight, 5MB each) and absorbs the multipart-completion plumbing. Bumps `@aws-sdk/client-s3` from `^3.700.0` to `^3.1050.0` to satisfy lib-storage's peer range. Both surfaces share `assertKey` and wrap unexpected errors in `ExternalServiceError`.

What landed:

- **`@dankdash/realtime-events` package** (`packages/realtime-events/`) — single source of truth for the wire format on the shared `dankdash:realtime` Redis Stream. Exports a `RealtimeEnvelope` zod schema (`{id, emittedAt, source, event}`) wrapping a discriminated `RealtimeEvent` union (`order:status_changed`, `order:driver_assigned`, `driver:location`, `vendor:notification`). `publishRealtimeEvent` XADDs an `envelope`-keyed JSON field; `decodeStreamEntry` validates back to typed shape so the consumer can switch on `event.type` exhaustively. JWT auth is RS256 against the same public key the API issues: `RealtimeJwtVerifier` decodes a base64 PEM once at boot, verifies iss/aud/sub/role/sid, and stashes `{token, claims}` on `socket.data`. On accept it publishes a `driver:location` envelope to the realtime stream — the only place an unprivileged client writes into the bus.
- **`StreamConsumer`** (`apps/realtime/src/streams/`) — one consumer group (`realtime` by default), one consumer name per pod (`hostname-pid`). The loop is `XPENDING IDLE … XCLAIM` recovery first (entries from a dead pod get picked up by survivors), then `XREADGROUP BLOCK 5000` for new entries; each entry decodes through `decodeStreamEntry`, routes via the pure `routeEnvelope` switch to `{namespace, room, eventName, payload}` broadcasts, then XACKs. Decode failures are ACKed (poison entries get logged once, not retried forever). `stop()` flips a running flag and awaits the in-flight BLOCK so SIGTERM redeploys are clean.
- **Two Redis connections, not one** (`apps/realtime/src/server.ts`) — the @socket.io/redis-adapter installs a `dankdash:io:` key prefix on the pubClient that would silently shift our application stream key, so the service builds a separate `streamClient = new Redis(REDIS_URL)` for XADD/XREADGROUP. And because ioredis serializes commands per connection, `XREADGROUP BLOCK 5000` on a shared connection would deadlock the producer's XADD for 5s — `consumerClient = streamClient.duplicate()` gives the blocking consumer its own connection. Both gotchas root-caused live during Phase 9.5 (initially every `driver:location:update` test hung silently); the comments in `server.ts` document the invariants so the next person doesn't re-derive them under fire.
- **Composition root + bootstrap** — `buildServer({env, logger, pool?, membership?})` is the single factory: no global singletons, no import-time side effects, returns `{app, httpServer, io, listen, close}`. Tests inject an in-memory membership and skip Postgres entirely. `close()` shuts the consumer first, then `io.close()` (which also tears down the engine.io-attached httpServer — the explicit `httpServer.close()` is guarded behind `httpServer.listening` to avoid ERR_SERVER_NOT_RUNNING). `main.ts` is the process entrypoint — loads env, builds pino, constructs the `@dankdash/db` pool, calls `buildServer`, and binds SIGTERM/SIGINT to `server.close()` for graceful redeploys.

Coverage: `auth.test.ts` (13 — valid tokens across all three roles, missing/expired/wrong-issuer/role-mismatch rejection, staff/driver membership enforcement, stream→room broadcasts), `rate-limit.test.ts` (5 — happy path, token-bucket starvation, zod validation, heartbeat ack, direct XLEN/XREVRANGE wire-format inspection), `streams.test.ts` (4 — order-status routing, vendor-notification routing, malformed envelope swallowed).

- **`dispatch_failed` terminal state** — adds the operational-failure sink for "no driver in the polygon picked up within the offer window" so an order never sits in `awaiting_driver` indefinitely. Migration `0006_dispatch_failed` adds the enum value; migration `0007_dispatch_failed_column` adds the `dispatch_failed_at` timestamptz column and widens `orders_active_idx` (the partial index over hot orders) to exclude the new terminal value. The split exists because Postgres rejects USE of a newly-added enum value in the same transaction that ADDed it (error 55P04 — `new enum values must be committed before they can be used`); 0007 references the value once 0006 has committed. The 0006 down-migration is a documented no-op — Postgres has no `ALTER TYPE … DROP VALUE`. `STATUS_TIMESTAMP_COLUMN` in `packages/db/src/repositories/orders.repo.ts` learns the new state so the transition service stamps `dispatch_failed_at` automatically. `@dankdash/orders` adds the state to `ORDER_STATES` (between `awaiting_driver` and `driver_assigned`) and to `TERMINAL_ORDER_STATES`; `events.ts` adds `DISPATCH_FAILED`; the XState machine accepts `awaiting_driver → dispatch_failed` from the system actor. - **`@dankdash/dispatch` package** (`packages/dispatch/`) — pure scoring + attempt orchestration with no NestJS / no Drizzle / no HTTP. `scoring.ts` exposes `scoreDriver(candidate, dispensary, now)` returning a non-negative integer composite of three weighted sub-scores: `distanceScore` (1 − `min(distanceMeters, RADIUS_METERS) / RADIUS_METERS`) × `DISTANCE_WEIGHT (0.6)`, `ratingScore` (driver rating normalized to `[0,1]` over `RATING_FLOOR=3.0..RATING_CEILING=5.0`) × `RATING_WEIGHT (0.25)`, and `recencyScore` (older `last_completed_delivery_at` ranks higher to avoid starving newer drivers; saturates at `RECENCY_HORIZON_SECONDS=21600` = 6 hours) × `RECENCY_WEIGHT (0.15)`. - **`@dankdash/orders` package** (`packages/orders/`) — pure state-machine package with no NestJS / no Drizzle / no HTTP. `states.ts` declares `ORDER_STATES` as a `readonly` tuple (19 states) and `TERMINAL_ORDER_STATES` (6: `payment_failed`, `rejected`, `delivered`, `returned_to_store`, `canceled`, `disputed`) — the tuple mirrors the `order_status` Postgres enum and the iOS `OrderStatus` enum, and an integration mirror test in `apps/api/test/integration/orders/` will catch drift. `order-machine.ts` builds the XState v5 machine via `setup({ types: { events: {} as OrderMachineEvent } }).createMachine(...)` so every event payload is typed at the call site. The machine is intentionally a pure transition table — no actions, no entry/exit side effects, no actors — so the `OrderTransitionService` can consult it inside a DB transaction without paying actor-spawn overhead. Public `nextOrderState(from, event)` runs a single-shot actor and returns the next state (or throws `OrderError.invalidTransition` with stable code `ORDER_INVALID_TRANSITION`). `isTerminalOrderState` short-circuits any event against a closed-final state before the machine spins up. `errors.ts` defines the `OrderError` discriminated subtype of `DomainError` with codes `ORDER_NOT_FOUND` (404), `ORDER_INVALID_TRANSITION` (422), `ORDER_ACTOR_FORBIDDEN` (403), `ORDER_TERMINAL_STATE` (422), and `ORDER_INVARIANT_BROKEN` (500). `events.ts` declares `OrderEventType` (a closed string union covering every edge — `VENDOR_ACCEPT`, `CUSTOMER_CANCEL`, `DRIVER_PICKED_UP`, `ID_SCAN_PASSED`, …) so adding an event there forces a matching case in the authz matrix and the per-state timestamp lookup.

- **`OrdersRepository.applyTransition(orderId, resolver)`** (`packages/db/src/repositories/orders.repo.ts`) — the new authoritative entry point for every status change. Opens a transaction, runs `SELECT … FOR UPDATE` on the order row, hands the locked snapshot to the caller's resolver, and the resolver returns a `TransitionDecision { toStatus, eventType, actorUserId?, actorRole?, payload?, reason?, patch? }`. With that decision the repo UPDATEs `orders.status` + `status_changed_at` + the per-state timestamp column derived from `STATUS_TIMESTAMP_COLUMN` (the lookup ensures the column name is never hand-rolled at the call site), spreads any `patch` fields (driverId, canceledBy, cancelReason, deliveryIdScan\*), INSERTs the immutable `order_events` row, INSERTs the typed `order_status_history` row — all inside the same tx. Throws `NotFoundError` if the row vanishes; `RepositoryError` on impossible "UPDATE … RETURNING returned no row" cases. The legacy `transitionStatus` method is marked `@deprecated` and remaining call sites have been migrated.
- **`OrderTransitionService`** (`apps/api/src/modules/orders/order-transition.service.ts`) — the **single chokepoint** for every UPDATE to `orders.status` and every INSERT into `order_events` / `order_status_history`. No other code path in the API is allowed to mutate order state; this is documented in the file header. The service runs an exhaustive `AUTH_BY_EVENT: Readonly<Record<OrderEventType, AuthFn>>` matrix where `Record<OrderEventType, …>` forces a compile-time match for every event in `@dankdash/orders/events.ts` — adding an event without an authz rule breaks `tsc`. Predicates: `vendorOwnsOrder` (vendor matches `dispensary_id` from the locked row + admin override), `customerOwnsOrder`, `assignedDriver` (driver row.driver_id matches the principal user_id), `allowSystemOrAdmin` (webhook + cron paths). Authorization runs **inside the lock** so a concurrent VENDOR_ACCEPT race sees the post-write status and bails with `ORDER_INVALID_TRANSITION` rather than silently double-writing. Defense-in-depth: a terminal state (except `delivered` → `disputed`) is rejected with `ORDER_TERMINAL_STATE` before the machine is consulted. The `fromStatus` is captured into a holder object (`{ value?: OrderState }`) so the post-commit unwrap is an explicit `OrderError.invariantBroken` throw rather than a non-null assertion — the assertion would have been "unreachable" but the holder pattern makes the unreachability the type checker's concern, not a TS-eslint suppression. Post-commit, the service emits `OrderTransitionedEvent` via EventEmitter2; if a subscriber throws, it is logged and swallowed — the DB has already committed, the caller's mutation is durable, and a downstream failure must not surface a 500 that suggests rollback.
- **`OrderTransitionedEvent` + EventEmitter2 plumbing** (`apps/api/src/modules/orders/order-transition.events.ts`) — typed event class carrying `{ orderId, fromStatus, toStatus, event, actor: { userId?, role, dispensaryId? }, occurredAt }`. `ORDER_TRANSITIONED_EVENT` constant is the emit key. `EventEmitterModule.forRoot({ wildcard: true, delimiter: '.', maxListeners: 20, verboseMemoryLeak: false })` is registered globally in `AppModule` so every feature module can publish/subscribe without redundant per-module imports. Subscribers consume the typed event class, not the raw payload — so a future schema change surfaces as a TS error in every subscriber. ## Phase 6 — Payments & Aeropay

_Status: complete (2026-05-18). Branch: `phase/06-payments`._

What landed:

- **`@dankdash/aeropay` package** (`packages/aeropay/`) — typed client for Aeropay's bank-link / payment / refund / payout APIs. OAuth2 client-credentials flow with a pluggable token cache (in-memory default, Redis impl in `apps/api`) so two API replicas don't stampede the token endpoint. Every mutation accepts an `idempotencyKey` and forwards it as the `Idempotency-Key` header — required for the payment / payout retry contract upstream. All request + response bodies validated by Zod schemas; a single `AeropayError` discriminated union carries `code` + `statusCode` + the parsed upstream error so callers can decide retry vs surface-to-user. Webhook HMAC verifier (`verifyWebhookSignature`) reads `t=<unix>,v1=<hex>` from the `Aeropay-Signature` header, recomputes `HMAC-SHA256(secret, "${t}.${rawBody}")`, rejects on ±5min skew, and uses `timingSafeEqual` on the digest compare so signature checks don't leak timing. 100% line coverage. Public surface is the barrel — deep imports forbidden.
- **`PaymentMethodsModule`** (`apps/api/src/modules/payments/`) — `GET /v1/payment-methods` lists the caller's saved methods (bank account refs only, never the raw account number), `POST /v1/payment-methods/aeropay/link` exchanges a short-lived link token for a stored bank-account reference, `DELETE /v1/payment-methods/:id` soft-removes. The Redis token cache lives here (`redis-token-cache.ts`) and is wired into the Aeropay client via DI; the `RedisTokenCache` writes with `SET ... EX` so expired tokens are GC'd by Redis rather than a sweeper job.
- **Payment lifecycle wired into the existing checkout transaction** — the Phase 5 stub `pi_stub_<shortCode>` is replaced by a real `aeropay.createPayment` call inside the checkout txn, keyed by `payment:<orderId>` so a network retry coalesces upstream. `payment_transactions.provider_ref` now stores the real Aeropay payment id; `status` flows `initiated` → `authorized` → `settled` (or `failed`) driven by the webhook. On `payment.settled` the distribution ledger is written: customer-receivable DR cleared by aeropay_clearing CR for the placement leg, then the settlement-clearing pair + the distribution pair (dispensary CR for share, platform_fee CR for the 15% take, driver CR for tip, gross-receipts tax CR, sales tax CR, refund_reserve CR for 2.5% of dispensary share). Total per-order DRs after settle = `3 × orderTotal` (placement + settlement-clearing + distribution). Settlement formula corrected vs spec §6.4: the dispensary share line subtracts discount in addition to platform fee + driver tip + tax, otherwise double-entry breaks — guarded by memory `settlement_dispensary_share.md`.
- **`AeropayWebhookController`** (`POST /v1/payment-methods/aeropay/webhook`) — raw-body capture via a Fastify content-type parser registered for `application/json` so the HMAC verifier sees byte-exact input (the JSON the auto-parser produces is not a stable serialization). Public route (no JWT — Aeropay calls it), per-IP rate-limited. Forged signature → 401, body-not-JSON → 400. Idempotency via `webhook_events_processed` (Phase 6.7): every successful dispatch inserts `(event_id, event_type, processed_at)` inside the same txn that mutates `payment_transactions` + writes the ledger; a replay finds the row and logs `webhook replay ignored` without re-running any side effect.
- **`webhook_events_processed` table** (`packages/db/src/migrations/0010_webhook_events_processed.sql`) — `event_id` PK, `event_type` text, `processed_at` timestamptz default `NOW()`. The Phase 6.7 cleanup cron (`apps/workers/src/jobs/webhook-events/cleanup.job.ts`) runs at 02:00 America/Chicago daily and deletes rows older than 30 days — Aeropay's replay window is 7 days, so 30d is a safe operational floor.

Per dispensary: gross = sum of CR entries on `account_type='dispensary'` in window; refund draw = sum of DR entries on `account_type='refund_reserve'` in window; net = gross − refund draw. If `aeropay_account_ref IS NULL` → payout row inserted with `status='failed'`, `failure_reason='dispensary_bank_account_not_linked'`, no Aeropay call.
