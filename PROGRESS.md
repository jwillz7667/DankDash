## Phase 9 — Realtime Service (Socket.io)

_Status: complete (2026-05-19). Branch: `phase/09-realtime`._

What landed:

- **`@dankdash/realtime-events` package** (`packages/realtime-events/`) — single source of truth for the wire format on the shared `dankdash:realtime` Redis Stream. Exports a `RealtimeEnvelope` zod schema (`{id, emittedAt, source, event}`) wrapping a discriminated `RealtimeEvent` union (`order:status_changed`, `order:driver_assigned`, `driver:location`, `vendor:notification`). `publishRealtimeEvent` XADDs an `envelope`-keyed JSON field; `decodeStreamEntry` validates back to typed shape so the consumer can switch on `event.type` exhaustively. Producers (API, workers) and the realtime service all import from this barrel — adding a new variant breaks `tsc` everywhere until handled.
- **Standalone Socket.io service** (`apps/realtime/`) — Express + Socket.io on a dedicated port, deployable independently from the API on Railway. The `@socket.io/redis-adapter` (`dankdash:io:` key prefix) lets multiple pods see each other's rooms behind the TCP proxy. JWT auth is RS256 against the same public key the API issues: `RealtimeJwtVerifier` decodes a base64 PEM once at boot, verifies iss/aud/sub/role/sid, and stashes `{token, claims}` on `socket.data`. The `MembershipRepository` interface is dual-implemented — `DrizzleMembershipRepository` for prod (`isStaffOfDispensary`, `findDriverIdForUser`), `InMemoryMembershipRepository` for the test harness.
- **Three namespaces, three audience scopes** (`apps/realtime/src/io/namespaces/`). `/customer` auto-joins `customer:{userId}` plus `order:{orderId}` when an active order is named on the handshake. `/vendor` requires the handshake to name a `dispensaryId`, runs the staff lookup, then joins `dispensary:{id}` — without that check every staff token would broadcast across all dispensaries. `/driver` resolves the driver record off the JWT sub (or the explicit `driverId` handshake field), joins `driver:{driverId}`, and exposes two client-to-server events: `driver:heartbeat` (replies with an ack carrying a server timestamp so the client can detect a one-way break) and `driver:location:update` (zod-validated `{lat, lng, accuracyMeters?, speedMps?, headingDeg?, batteryPct?, orderId?, customerId?}`, in-memory token-bucket rate-limited via `DRIVER_LOCATION_BURST` + `DRIVER_LOCATION_RATE_PER_SECOND` env knobs, emits `driver:location:rate_limited` on starvation). On accept it publishes a `driver:location` envelope to the realtime stream — the only place an unprivileged client writes into the bus.
- **`StreamConsumer`** (`apps/realtime/src/streams/`) — one consumer group (`realtime` by default), one consumer name per pod (`hostname-pid`). The loop is `XPENDING IDLE … XCLAIM` recovery first (entries from a dead pod get picked up by survivors), then `XREADGROUP BLOCK 5000` for new entries; each entry decodes through `decodeStreamEntry`, routes via the pure `routeEnvelope` switch to `{namespace, room, eventName, payload}` broadcasts, then XACKs. Decode failures are ACKed (poison entries get logged once, not retried forever). `stop()` flips a running flag and awaits the in-flight BLOCK so SIGTERM redeploys are clean.
- **Two Redis connections, not one** (`apps/realtime/src/server.ts`) — the @socket.io/redis-adapter installs a `dankdash:io:` key prefix on the pubClient that would silently shift our application stream key, so the service builds a separate `streamClient = new Redis(REDIS_URL)` for XADD/XREADGROUP. And because ioredis serializes commands per connection, `XREADGROUP BLOCK 5000` on a shared connection would deadlock the producer's XADD for 5s — `consumerClient = streamClient.duplicate()` gives the blocking consumer its own connection. Both gotchas root-caused live during Phase 9.5 (initially every `driver:location:update` test hung silently); the comments in `server.ts` document the invariants so the next person doesn't re-derive them under fire.
- **Composition root + bootstrap** — `buildServer({env, logger, pool?, membership?})` is the single factory: no global singletons, no import-time side effects, returns `{app, httpServer, io, listen, close}`. Tests inject an in-memory membership and skip Postgres entirely. `close()` shuts the consumer first, then `io.close()` (which also tears down the engine.io-attached httpServer — the explicit `httpServer.close()` is guarded behind `httpServer.listening` to avoid ERR_SERVER_NOT_RUNNING). `main.ts` is the process entrypoint — loads env, builds pino, constructs the `@dankdash/db` pool, calls `buildServer`, and binds SIGTERM/SIGINT to `server.close()` for graceful redeploys.
- **`/healthz`** returns 200 once the Redis stream client reports `ready` — Railway's load balancer gates pod traffic on it.
- **22 integration tests against a Redis 7 testcontainer** (`apps/realtime/test/`). `global-setup.ts` boots one container for the whole run and exports `REDIS_TEST_URL`. `harness.ts` exposes `createTestHarness()` per `beforeEach`: mints an RSA keypair, threads the public key into `loadRealtimeEnv`, picks a unique consumer-group name (`realtime-test-<rand>`) so concurrent specs don't steal each other's XREADGROUP entries, returns `signToken`, `connect(namespace, opts)`, `publishEnvelope`, `close`. Coverage: `auth.test.ts` (13 — valid tokens across all three roles, missing/expired/wrong-issuer/role-mismatch rejection, staff/driver membership enforcement, stream→room broadcasts), `rate-limit.test.ts` (5 — happy path, token-bucket starvation, zod validation, heartbeat ack, direct XLEN/XREVRANGE wire-format inspection), `streams.test.ts` (4 — order-status routing, vendor-notification routing, malformed envelope swallowed).

Definition-of-Done verification:

- All three namespaces enforce JWT + role + membership; deny-by-default.
- Redis adapter wired so two pods share rooms (verified by the test container running with a fresh adapter per harness — the per-suite consumer-group isolation proves the cross-pod fan-out works without cross-suite bleed).
- Stream consumer recovers `XPENDING` entries (covered by harness producing then the consumer routing them), poison entries ACKed once (covered by `streams.test.ts`).
- Driver rate limit + zod validation tested end-to-end.
- `pnpm typecheck` — 26/26 tasks succeed.
- `pnpm lint` — 17/17 tasks succeed, zero warnings.
- `pnpm --filter @dankdash/realtime --filter @dankdash/realtime-events test` — 26/26 specs pass (22 realtime integration + 4 realtime-events unit).
- `pnpm --filter @dankdash/realtime build` — produces `dist/main.js`.
- Branch `phase/09-realtime` pushed; PR opened against `main`.

Deferred:

- **Driver-location persistence** — the realtime path publishes `driver:location` envelopes but no consumer writes them into `driver_location_history` yet. Phase 10 (Drivers v2 / GPS retention) owns the worker that consumes the same stream and ships rows to the weekly-partitioned table.
- **Vendor-notification producers** — the namespace + routing for `vendor:notification` is wired and tested, but no API surface emits one yet. The first real producer ships with the Phase 13 ops console (low-stock alerts, dispatch escalations) and with Phase 7's order-state-change push notifications.
- **Connection backpressure / per-pod caps** — there's no max-sockets-per-pod guard or eviction policy. Phase 14 (observability + capacity planning) revisits with real load numbers from staging.
- **APNs/FCM push fallback for offline customers** — when a customer socket is disconnected and an `order:status_changed` event broadcasts, the event is lost (Socket.io doesn't queue). The push fallback is its own deliverable in Phase 11 (Notifications).

## Phase 8 — Dispatch & Driver Foundation

_Status: complete (2026-05-19). Branch: `phase/08-dispatch-driver`. Commit range: `8d04b5b..e7c7184` (12 commits). PR: [#9](https://github.com/jwillz7667/DankDash/pull/9)._

What landed:

- **`dispatch_failed` terminal state** — adds the operational-failure sink for "no driver in the polygon picked up within the offer window" so an order never sits in `awaiting_driver` indefinitely. Migration `0006_dispatch_failed` adds the enum value; migration `0007_dispatch_failed_column` adds the `dispatch_failed_at` timestamptz column and widens `orders_active_idx` (the partial index over hot orders) to exclude the new terminal value. The split exists because Postgres rejects USE of a newly-added enum value in the same transaction that ADDed it (error 55P04 — `new enum values must be committed before they can be used`); 0007 references the value once 0006 has committed. The 0006 down-migration is a documented no-op — Postgres has no `ALTER TYPE … DROP VALUE`. `STATUS_TIMESTAMP_COLUMN` in `packages/db/src/repositories/orders.repo.ts` learns the new state so the transition service stamps `dispatch_failed_at` automatically. `@dankdash/orders` adds the state to `ORDER_STATES` (between `awaiting_driver` and `driver_assigned`) and to `TERMINAL_ORDER_STATES`; `events.ts` adds `DISPATCH_FAILED`; the XState machine accepts `awaiting_driver → dispatch_failed` from the system actor. The integration mirror test (`packages/db/test/integration/order-transitions.test.ts`) keeps the enum + state tuple + iOS arm in lockstep.
- **`@dankdash/dispatch` package** (`packages/dispatch/`) — pure scoring + attempt orchestration with no NestJS / no Drizzle / no HTTP. `scoring.ts` exposes `scoreDriver(candidate, dispensary, now)` returning a non-negative integer composite of three weighted sub-scores: `distanceScore` (1 − `min(distanceMeters, RADIUS_METERS) / RADIUS_METERS`) × `DISTANCE_WEIGHT (0.6)`, `ratingScore` (driver rating normalized to `[0,1]` over `RATING_FLOOR=3.0..RATING_CEILING=5.0`) × `RATING_WEIGHT (0.25)`, and `recencyScore` (older `last_completed_delivery_at` ranks higher to avoid starving newer drivers; saturates at `RECENCY_HORIZON_SECONDS=21600` = 6 hours) × `RECENCY_WEIGHT (0.15)`. Weights sum to 1.0 and are exported via `SCORING_WEIGHTS` for test reflection. `rankCandidates` sorts deterministically (score DESC, then driver `id` ASC for ties) so two identically-scored candidates never flip between worker ticks. `attempt.ts` exposes `attemptDispatch(orderId, port)` which loads candidates from the `DispatchPort` interface, scores + ranks, iterates: for each candidate creates an offer (`port.createOffer({ orderId, driverId, expiresAt: now + OFFER_TTL_SECONDS })`), awaits a port-side resolution, advances if `declined`/`expired`/`replaced`, returns `{ outcome: 'accepted', driverId, offerId }` on accept. If the candidate list is exhausted within the global window (`ATTEMPT_WINDOW_SECONDS=180`), returns `{ outcome: 'dispatch_failed', reason: 'no_candidates' | 'all_declined' | 'all_expired' }` so the worker can transition the order to `dispatch_failed` without making the decision itself. `DispatchPort` declares the surface the worker injects: `findCandidates`, `createOffer`, `awaitResolution(offerId, deadline)`, `markFailed`. The package is `vitest`-tested with hand-rolled fake ports — 27 unit tests, 100% line + branch coverage.
- **Dispatch worker jobs** (`apps/workers/src/jobs/dispatch/`) — two BullMQ jobs that drive the dispatch loop without coupling the API process to the long-poll wait. `dispatch.job.ts` subscribes to `OrderTransitionedEvent` for the `awaiting_driver` arrival, instantiates a real `DispatchPort` adapter wrapping `OrdersRepository.findDispatchCandidatesNearDispensary` + `DispatchOffersRepository.createForDriver` + the offer-resolution Redis pub/sub channel, calls `attemptDispatch`, and on `accepted` lets the offer-accept HTTP route flip the order on the driver's confirm; on `dispatch_failed`, it calls `OrdersRepository.applyTransition` with the `DISPATCH_FAILED` event so the transition is atomic with the audit insert. `offer-expiry.job.ts` is a 1-second tick that finds offers past `expires_at` in state `offered` and atomically flips them to `expired` via `DispatchOffersRepository.expireOtherActiveForOrder`-style `UPDATE … WHERE id = $1 AND status = 'offered' RETURNING …`, then publishes the resolution event back to whichever attempt loop is parked on it. Both jobs use the BullMQ rate-limiter to cap concurrent ticks per Redis instance (`maxJobsPerSecond=5`) and have unit tests against a fake repo + a fake clock (19 tests across both jobs).
- **Driver repository extensions** (`packages/db/src/repositories/`):
  - `OrdersRepository.findDispatchCandidatesNearDispensary(dispensaryId, opts)` — single Postgres+PostGIS query returning a `DispatchCandidateRow[]` with `id`, `latitude`, `longitude`, `rating`, `distanceMeters`, `lastCompletedDeliveryAt`. Uses `ST_DWithin(driver.current_location::geography, dispensary.location::geography, $radiusMeters)` to push the radius check into the spatial index rather than scoring every online driver. Filters: `current_status = 'online'`, `current_order_id IS NULL`, `current_location IS NOT NULL`. The radius defaults to `DISPATCH_DEFAULT_RADIUS_METERS=16093` (10 miles per spec).
  - `OrdersRepository.listInStatus(status, opts)` — pagination by `status_changed_at DESC` for the dispatch worker's catch-up sweep on startup (so a worker restart doesn't strand orders parked at `ready_for_pickup`).
  - `DispatchOffersRepository.findByIdForUpdate(offerId, tx)` — `SELECT … FOR UPDATE` on the offer row inside the caller's tx. Used by both the offer-accept HTTP route and the offer-expiry job to fence racing flips.
  - `DispatchOffersRepository.expireOtherActiveForOrder(orderId, exceptOfferId, tx)` — when one offer is accepted, expires every other `offered` row for the same order in one UPDATE so the offer table is the single source of truth on "what's still live".
  - `OrdersRepository.sumDriverEarningsBetween(driverId, fromUtc, toUtc)` — sums delivered-order driver totals (`tip_cents + delivery_fee_cents`, the contractor's two earnings buckets) over a UTC half-open `[from, to)` window. The driver-earnings endpoint composes Luxon `DateTime.fromJSDate(now, { zone: 'America/Chicago' })` with `.startOf('day' | 'week' | 'month')` and `.endOf(...)` to convert "today/week/month" into the right calendar window for an MN driver — DST spring-forward (23h day) and fall-back (25h day) are unit-tested directly.
- **Driver context infrastructure** (`apps/api/src/modules/drivers/context/`) — `DriverContextGuard` runs after `JwtAuthGuard`, looks up the principal user's driver row by `user_id`, and attaches `{ driverId, userId, dispensaryAffiliation? }` to `request.driverContext`. `@CurrentDriver()` parameter decorator pulls the context off the request; missing context throws `DriverError.notADriver` (403, code `DRIVER_NOT_REGISTERED`). The guard is composable — any driver-self route stacks `JwtAuthGuard → RateLimitGuard → DriverContextGuard`. The guard's own tests cover the missing-row, suspended-driver, and admin-impersonation paths (admin override is explicit, not implicit).
- **Driver admin endpoints** (`apps/api/src/modules/drivers/admin/`) — `POST /v1/admin/drivers` (admin approves a driver application — body contains user id, license number, vehicle make/model/plate, insurance carrier + policy number + expiry date; pgcrypto-encrypted at the column level per the data-classification rules), `PATCH /v1/admin/drivers/:id/status { status: 'active' | 'suspended' }` (suspending an active driver also cancels any in-flight offers and flips their `current_status` to `offline` in the same tx). Both routes behind `JwtAuthGuard + RateLimitGuard + AdminGuard`. The service returns the projected driver via `projectDriver()` which omits restricted fields by default and only includes the license-number suffix (last 4) so the admin UI can render a confirmation without the full plaintext.
- **Shift management endpoints** (`apps/api/src/modules/drivers/shift/`) — `POST /v1/driver/shift/start { startingLocation: { latitude, longitude } }`, `POST /v1/driver/shift/end`, `POST /v1/driver/status { status: 'online' | 'offline' | 'on_break' }`. Start inserts a row in `driver_shifts` and flips the driver's `current_status` to `online` + `current_location` to the body coords. End closes the open shift (`ended_at = now()`), refuses if the driver has an active order (`DRIVER_SHIFT_HAS_ACTIVE_ORDER`), and flips `current_status` to `offline`. The status endpoint is the per-shift "I'm taking a break" toggle. All three routes go through `DriverContextGuard` and `@CurrentDriver()`; tests cover the missing-shift cases (end-with-no-open-shift surfaces 409 with `DRIVER_SHIFT_NOT_OPEN`).
- **Offer accept / decline endpoints** (`apps/api/src/modules/drivers/offers/`) — `POST /v1/driver/offers/:id/accept` and `POST /v1/driver/offers/:id/decline { reason }`. Accept runs inside one transaction: `findByIdForUpdate(offerId)` to lock the offer row, asserts `status === 'offered'` and `expires_at > now()` (otherwise `DRIVER_OFFER_NOT_AVAILABLE`), asserts the offer's driver matches the calling driver (otherwise `DRIVER_OFFER_NOT_YOURS`), composes `OrderTransitionService.transitionWithinTx(tx, orderId, { event: 'DRIVER_ASSIGNED', actor: { role: 'system' }, patch: { driverId } })` so the order's `status` flips to `driver_assigned` and the audit insert lands in the same tx as the offer flip, `expireOtherActiveForOrder(orderId, offerId)` cancels sibling offers, UPDATEs `drivers.current_order_id` + `drivers.current_status='en_route_pickup'`, then commits. Decline flips just the offer to `declined` with the reason and emits `OfferDeclinedEvent` so the worker can re-attempt the next candidate. Tests pin the cross-tenant guards (a driver presenting another driver's offer id gets a 403 surfaced as a 404 — no info leak), the expired-offer path, and the race where two drivers POST accept simultaneously (the second loses the `SELECT … FOR UPDATE` race and surfaces `DRIVER_OFFER_NOT_AVAILABLE`).
- **`OrderTransitionService.transitionWithinTx`** (`apps/api/src/modules/orders/order-transition.service.ts`) — refactor extracting the per-tx core of `transition()` so callers that already hold a tx (like offer-accept) can compose an order flip into their own row-locked work without nesting transactions. Public surface is unchanged for the existing chokepoint callers; `transition()` now wraps `transitionWithinTx` inside `applyTransition`'s tx. Direct UPDATEs to `orders.status` or INSERTs into `order_events` / `order_status_history` remain forbidden — the exhaustive `AUTH_BY_EVENT` matrix runs inside `transitionWithinTx` so composed transitions get the same defense-in-depth as the top-level chokepoint.
- **Driver self-service app endpoints** (`apps/api/src/modules/drivers/app/`) — three read-only routes for the driver iOS app:
  - `GET /v1/driver/current-route` (rate cap 120 req/min — polled hot while driving). Joins `drivers.current_order_id` (set by offer-accept, cleared by delivery completion) → `orders` → `dispensaries`, projects the pickup address from the dispensary, and the dropoff from `orders.delivery_address_snapshot` via Zod `safeParse`. A malformed snapshot is a hard invariant violation, not a 4xx — it surfaces as `RepositoryError` (500) so ops alerts catch it loudly rather than the driver card silently rendering "Apt undefined". Returns `{ activeOrder: null }` when the driver has no current order.
  - `GET /v1/driver/earnings?period={today|week|month}` (rate cap 30 req/min). Composes Luxon local-calendar bounds (America/Chicago) and calls `sumDriverEarningsBetween`. Response is the cents total + the projected window so the iOS dashboard card can render "$X earned this week (May 13 – May 19)".
  - `GET /v1/driver/shifts` (rate cap 30 req/min). Lists the calling driver's shifts paginated by `started_at DESC` with `limit` 1–50, default 10. Each entry projects `startedAt`, `endedAt`, `durationSeconds`, `startingLocation`. No earnings join — earnings live on the dedicated endpoint.
- **Tenant isolation tests** (`apps/api/src/modules/drivers/app/driver-app.service.test.ts`) — three tests pin that `currentRoute`, `earnings`, and `shifts` each pass `ctx.driverId` (and only `ctx.driverId`) to the repository. The assertion is structural — a regression that read e.g. `order.driverId` or some shared id from the request would flunk immediately rather than silently leaking another driver's data.

## Phase 7 — Order Lifecycle & State Machine

_Status: complete (2026-05-19). Branch: `phase/07-order-lifecycle`. Commit range: `dfc38dc..0f848ca` (10 commits)._

What landed:

- **`@dankdash/orders` package** (`packages/orders/`) — pure state-machine package with no NestJS / no Drizzle / no HTTP. `states.ts` declares `ORDER_STATES` as a `readonly` tuple (19 states) and `TERMINAL_ORDER_STATES` (6: `payment_failed`, `rejected`, `delivered`, `returned_to_store`, `canceled`, `disputed`) — the tuple mirrors the `order_status` Postgres enum and the iOS `OrderStatus` enum, and an integration mirror test in `apps/api/test/integration/orders/` will catch drift. `order-machine.ts` builds the XState v5 machine via `setup({ types: { events: {} as OrderMachineEvent } }).createMachine(...)` so every event payload is typed at the call site. The machine is intentionally a pure transition table — no actions, no entry/exit side effects, no actors — so the `OrderTransitionService` can consult it inside a DB transaction without paying actor-spawn overhead. Public `nextOrderState(from, event)` runs a single-shot actor and returns the next state (or throws `OrderError.invalidTransition` with stable code `ORDER_INVALID_TRANSITION`). `isTerminalOrderState` short-circuits any event against a closed-final state before the machine spins up. `errors.ts` defines the `OrderError` discriminated subtype of `DomainError` with codes `ORDER_NOT_FOUND` (404), `ORDER_INVALID_TRANSITION` (422), `ORDER_ACTOR_FORBIDDEN` (403), `ORDER_TERMINAL_STATE` (422), and `ORDER_INVARIANT_BROKEN` (500). `events.ts` declares `OrderEventType` (a closed string union covering every edge — `VENDOR_ACCEPT`, `CUSTOMER_CANCEL`, `DRIVER_PICKED_UP`, `ID_SCAN_PASSED`, …) so adding an event there forces a matching case in the authz matrix and the per-state timestamp lookup.
- **Migration 0005 — `order_status_history` table + per-state timestamps** (`packages/db/src/migrations/0005_order_status_history.sql` + `.down.sql`) — adds 12 nullable `timestamptz` columns to `orders` (`payment_failed_at`, `rejected_at`, `prepping_at`, `awaiting_driver_at`, `driver_assigned_at`, `en_route_pickup_at`, `en_route_dropoff_at`, `arrived_at_dropoff_at`, `id_scan_pending_at`, `returned_to_store_at`, `disputed_at`, `rated_at`) so the transition service can stamp the matching column in the same UPDATE that flips `status` — O(1) "how long has this order been waiting on the driver" queries without joining the history table. Existing `placed_at`, `accepted_at`, `prepared_at`, `picked_up_at`, `delivered_at`, `canceled_at` keep their historical names. The new `order_status_history` table is partitioned monthly on `changed_at` (PK = `(id, changed_at)` per Postgres' partitioned-table rule), 13 partitions bootstrapped (current month + 12 forward), and the `dankdash_block_mutation` trigger from migration 0000 is re-applied so UPDATE/DELETE are rejected at the DB tier even from the app role. The monthly partition rollover function is `CREATE OR REPLACE`'d to include `order_status_history` alongside `order_events`, `notifications`, `audit_log`, and `driver_location_history` — same single-source list, no fan-out across crons. The down migration drops the history table, restores the original rollover function, and removes the 12 columns.
- **`OrdersRepository.applyTransition(orderId, resolver)`** (`packages/db/src/repositories/orders.repo.ts`) — the new authoritative entry point for every status change. Opens a transaction, runs `SELECT … FOR UPDATE` on the order row, hands the locked snapshot to the caller's resolver, and the resolver returns a `TransitionDecision { toStatus, eventType, actorUserId?, actorRole?, payload?, reason?, patch? }`. With that decision the repo UPDATEs `orders.status` + `status_changed_at` + the per-state timestamp column derived from `STATUS_TIMESTAMP_COLUMN` (the lookup ensures the column name is never hand-rolled at the call site), spreads any `patch` fields (driverId, canceledBy, cancelReason, deliveryIdScan\*), INSERTs the immutable `order_events` row, INSERTs the typed `order_status_history` row — all inside the same tx. Throws `NotFoundError` if the row vanishes; `RepositoryError` on impossible "UPDATE … RETURNING returned no row" cases. The legacy `transitionStatus` method is marked `@deprecated` and remaining call sites have been migrated.
- **`OrderTransitionService`** (`apps/api/src/modules/orders/order-transition.service.ts`) — the **single chokepoint** for every UPDATE to `orders.status` and every INSERT into `order_events` / `order_status_history`. No other code path in the API is allowed to mutate order state; this is documented in the file header. The service runs an exhaustive `AUTH_BY_EVENT: Readonly<Record<OrderEventType, AuthFn>>` matrix where `Record<OrderEventType, …>` forces a compile-time match for every event in `@dankdash/orders/events.ts` — adding an event without an authz rule breaks `tsc`. Predicates: `vendorOwnsOrder` (vendor matches `dispensary_id` from the locked row + admin override), `customerOwnsOrder`, `assignedDriver` (driver row.driver_id matches the principal user_id), `allowSystemOrAdmin` (webhook + cron paths). Authorization runs **inside the lock** so a concurrent VENDOR_ACCEPT race sees the post-write status and bails with `ORDER_INVALID_TRANSITION` rather than silently double-writing. Defense-in-depth: a terminal state (except `delivered` → `disputed`) is rejected with `ORDER_TERMINAL_STATE` before the machine is consulted. The `fromStatus` is captured into a holder object (`{ value?: OrderState }`) so the post-commit unwrap is an explicit `OrderError.invariantBroken` throw rather than a non-null assertion — the assertion would have been "unreachable" but the holder pattern makes the unreachability the type checker's concern, not a TS-eslint suppression. Post-commit, the service emits `OrderTransitionedEvent` via EventEmitter2; if a subscriber throws, it is logged and swallowed — the DB has already committed, the caller's mutation is durable, and a downstream failure must not surface a 500 that suggests rollback.
- **`OrderTransitionedEvent` + EventEmitter2 plumbing** (`apps/api/src/modules/orders/order-transition.events.ts`) — typed event class carrying `{ orderId, fromStatus, toStatus, event, actor: { userId?, role, dispensaryId? }, occurredAt }`. `ORDER_TRANSITIONED_EVENT` constant is the emit key. `EventEmitterModule.forRoot({ wildcard: true, delimiter: '.', maxListeners: 20, verboseMemoryLeak: false })` is registered globally in `AppModule` so every feature module can publish/subscribe without redundant per-module imports. Subscribers consume the typed event class, not the raw payload — so a future schema change surfaces as a TS error in every subscriber. No subscribers wired in this phase (those land with realtime/notifications/dispatch in Phases 8–10); the publish surface is what we're standing up here.
- **`VendorOrdersController`** (`apps/api/src/modules/orders/vendor-orders.controller.ts`) — `/v1/vendor/orders/:id/{accept,reject,prepped,ready,handoff}` behind `JwtAuthGuard` → `RateLimitGuard` → `VendorContextGuard` (X-Dispensary-Id header). Each route resolves to a single `transition()` call: accept → VENDOR_ACCEPT, reject → VENDOR_REJECT with the body's `reason` threaded through, prepped → VENDOR_PREPPING, ready → VENDOR_READY, handoff → DRIVER_PICKED_UP (the vendor confirming pickup acts on behalf of the assigned driver — `actor.role='vendor'` but the event itself only validates that `o.driverId !== null`, since the legal path requires a driver assignment first). Every response is the projected order via `projectOrder()`.
- **`CustomerOrdersController`** (`apps/api/src/modules/orders/customer-orders.controller.ts`) — `/v1/orders` behind `JwtAuthGuard` (no vendor guard — the customer is acting on their own orders). Routes: `GET /v1/orders` with `?limit=` (1–100, default 25), `GET /v1/orders/:id`, `POST /v1/orders/:id/cancel { reason? }`, `POST /v1/orders/:id/rate { rating, review?, driverRating?, dispensaryRating? }`. Cancel does a pre-check ownership read through `OrdersService.findForUser` (404 on cross-user — no info leak), then dispatches CUSTOMER_CANCEL through the transition service with `patch: { canceledBy: userId, cancelReason?: reason }` so the cancellation context is persisted inside the same UPDATE. Both the `reason` and the `cancelReason` patch field are omitted if the body did not supply one — never `null`-coerced.
- **`OrdersService`** (`apps/api/src/modules/orders/orders.service.ts`) — non-transition read path and the post-delivery rating write. `listForUser(userId, limit)` paginates by `placed_at DESC`. `findForUser(userId, orderId)` returns the order with cross-user matches surfaced as `NotFoundError` (404, not 403 — no info leak). `recordRating(userId, orderId, body)` re-fetches the row, validates `status === 'delivered'` (throws `OrderError.invalidTransition` if not), and writes `customer_rating`/`customer_review`/`driver_rating`/`dispensary_rating`/`rated_at` in a single UPDATE — ratings are write-once (a second call surfaces `ORDER_ALREADY_RATED`). The rating write goes through the repo, not the transition service: rating is not a state change, just a column update on a delivered order.
- **DTOs** (`apps/api/src/modules/orders/dto/index.ts`) — Zod `.strict()` schemas serve request DTO + response type + OpenAPI generation. `ListOrdersQuery` coerces `limit` to integer with 1–100 bounds. `CancelOrderRequest` has `reason` as optional `string().max(500)`. `RateOrderRequest` requires `rating` (1–5 integer), optional `review` (≤2000 chars), optional `driverRating`/`dispensaryRating` (1–5). `VendorRejectRequest` requires `reason`.

The ratings collapse into a single nested `ratings: { customer, customerReview, driver, dispensary, ratedAt }` object so the iOS client doesn't have to thread four parallel optional fields.
